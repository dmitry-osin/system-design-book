**Глава 3. Базы данных: Введение**

Данные — это самое ценное, что есть в системе. Серверы (stateless) можно убивать и поднимать тысячами, но если упадет или испортится база данных — это катастрофа.

#### 1. Реляционные СУБД (RDBMS): Нормализация, Индексы и ACID

Реляционные базы (PostgreSQL, MySQL, Oracle) хранят данные в таблицах со строгой схемой. Они идеальны, когда между данными есть четкие связи (relations).

* **Нормализация:** Это процесс разбиения данных на несколько таблиц для устранения дублирования.
* *Пример:* Вместо того чтобы в таблице `Orders` каждый раз писать "Иванов Иван, Москва, ул. Пушкина", мы выносим это в таблицу `Users`, а в `Orders` храним только `user_id`. Это экономит место и спасает от рассинхронизации при обновлении адреса.


* **Индексы (B-Tree):** Без индекса база ищет строку, перебирая всю таблицу (Full Table Scan) — это $O(N)$. Индекс работает как оглавление в книге. В RDBMS индексы обычно строятся на структуре данных B-Tree (сбалансированное дерево), что дает скорость поиска $O(\log N)$.
* *Важное правило:* Чем больше индексов, тем быстрее чтение (`SELECT`), но тем медленнее запись (`INSERT`/`UPDATE`), потому что при каждой записи базе нужно перестраивать деревья индексов.


* **Транзакции (ACID):** Фундамент надежности, особенно в финтехе.
* **A**tomicity (Атомарность): Транзакция выполняется либо целиком, либо никак. (Перевод денег: списали у А, начислили Б. Если на этапе Б произошла ошибка, деньги А возвращаются).
* **C**onsistency (Согласованность): Транзакция не может нарушить правила базы (например, баланс не может стать отрицательным, если стоит ограничение `CHECK (balance >= 0)`).
* **I**solation (Изолированность): Параллельные транзакции не должны влиять друг на друга (подробнее ниже).
* **D**urability (Долговечность): Если база ответила "Успешно", данные не пропадут даже при отключении питания (достигается за счет записи лога операций — WAL: Write-Ahead Log).



#### 2. Уровни изоляции транзакций и феномены

В высоконагруженных системах тысячи транзакций происходят одновременно. Если их полностью изолировать, база будет работать медленно (выстроит всех в очередь). Поэтому базы позволяют "ослабить" изоляцию, но за это мы платим появлением аномалий (феноменов).

* **Dirty Read (Грязное чтение):** Транзакция 1 меняет баланс, но еще не делает `COMMIT`. Транзакция 2 читает этот новый баланс. Если Транзакция 1 сделает `ROLLBACK` (откат), то Транзакция 2 будет работать с несуществующими ("грязными") деньгами.
* **Non-Repeatable Read (Неповторяющееся чтение):** Транзакция 1 читает баланс (100$). В это время Транзакция 2 списывает 50$ и делает `COMMIT`. Транзакция 1 снова читает баланс в рамках *той же* транзакции и видит уже 50$. Данные изменились под ногами.
* **Phantom Read (Фантомное чтение):** Транзакция 1 запрашивает "всех VIP-клиентов" (их 10). Транзакция 2 создает нового VIP-клиента и коммитит. Транзакция 1 снова запрашивает VIP-клиентов и получает 11 записей. Появился "фантом".

**Уровни изоляции (от быстрого к надежному):**

1. **Read Uncommitted:** Возможны все феномены (почти нигде не используется).
2. **Read Committed:** (Дефолт в Postgres). Спасает от Dirty Read. Мы видим только закоммиченные данные.
3. **Repeatable Read:** (Дефолт в MySQL/InnoDB). Спасает от Dirty и Non-Repeatable Read. Если мы прочитали строку, никто не сможет ее изменить, пока мы не завершим транзакцию.
4. **Serializable:** Полная изоляция. Транзакции выполняются так, как будто они идут строго по очереди. Самый медленный, но самый безопасный уровень.

**Пример на Java (Spring Boot & Hibernate):**
В Spring мы управляем этим через аннотацию `@Transactional`. В финтехе при начислении процентов часто требуется строгая изоляция.

```java
import org.springframework.transaction.annotation.Transactional;
import org.springframework.transaction.annotation.Isolation;
import org.springframework.stereotype.Service;

@Service
public class AccountService {

    // Включаем максимальный уровень изоляции. 
    // Это гарантирует, что никто не добавит "фантомные" транзакции 
    // и не изменит баланс, пока мы проводим аудит и начисление.
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void calculateAndApplyInterest(Long accountId) {
        // 1. Читаем текущий баланс
        Account account = accountRepository.findById(accountId).orElseThrow();
        
        // 2. Сложная логика расчетов
        double interest = account.getBalance() * 0.05; 
        
        // 3. Обновляем баланс
        account.setBalance(account.getBalance() + interest);
        accountRepository.save(account);
        
        // В случае ошибки на любом этапе, Spring автоматически сделает ROLLBACK.
    }
}

```

#### 3. NoSQL: Когда таблиц недостаточно

Когда данных становится петабайты, или структура меняется каждый день, RDBMS начинают тяжело дышать. На помощь приходят NoSQL-решения. Они жертвуют ACID-гарантиями или сложными связями ради горизонтального масштабирования и скорости.

* **Key-Value (Redis, Memcached):** Данные хранятся как словарь. Идеально для кэширования, хранения сессий пользователей, корзин покупок. Работает в оперативной памяти (RAM) — скорости невероятные (миллисекунды).
* **Document (MongoDB, Couchbase):** Хранят данные в формате, похожем на JSON (BSON).
* *Пример:* Каталог товаров в маркетплейсе. У ноутбука есть `processor`, а у футболки `size`. В SQL пришлось бы делать кучу таблиц для атрибутов. В Mongo мы просто кидаем JSON с произвольными полями в одну коллекцию `Products`.


* **Column-family (Cassandra, HBase):** Хранят данные не по строкам, а по колонкам. Оптимизированы для колоссально быстрой записи.
* *Пример:* Сбор логов, метрик с IoT-устройств, история сообщений в мессенджере (Discord начинал с Cassandra). Они могут переваривать миллионы записей в секунду.


* **Graph (Neo4j):** Фокус не на самих данных, а на **связях** между ними.
* *Пример:* Социальная сеть ("найди друзей друзей моих друзей"), системы рекомендаций, антифрод (поиск мошеннических колец: устройство A связано с картой B, которая связана с IP-адресом C).



#### 4. SQL vs NoSQL: Критерии выбора

На собеседовании по System Design первый вопрос при выборе базы должен быть: "Какие у нас данные и как мы будем их читать/писать?"

* **Выбирай SQL (Postgres/MySQL), если:**
* Данные строго структурированы.
* Требуются сложные запросы (JOIN'ы нескольких сущностей).
* Нужны строгие гарантии ACID (финансы, биллинг, инвентаризация).


* **Выбирай NoSQL, если:**
* Данных очень много (Big Data) и их нужно распределять по десяткам серверов (Scale Out).
* Схема данных постоянно меняется (Agile).
* Нагрузка в основном на запись (Write-heavy) -> *Cassandra*.
* Нужен сверхбыстрый доступ по ключу -> *Redis*.



---

**Итог:**

* Реляционные базы — это надежность (ACID), структура (нормализация) и быстрый поиск за счет индексов (B-Tree). Главная боль — сложность масштабирования на множество серверов.
* Изоляция транзакций — это компромисс между скоростью работы базы и защитой от аномалий (Грязных, Неповторяющихся и Фантомных чтений).
* NoSQL базы решают узкоспециализированные задачи: Redis для кэша, Mongo для гибких схем (документов), Cassandra для огромного потока записи, Neo4j для графов связей.
* В современных энтерпрайз-архитектурах используют подход **Polyglot Persistence**: основная бизнес-логика и деньги лежат в PostgreSQL, сессии — в Redis, логи и аналитика — в Cassandra.
