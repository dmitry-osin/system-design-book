**Глава 18. Наблюдаемость (Observability)**

Когда в монолите происходила ошибка, ты просто открывал файл `app.log` на сервере и читал `StackTrace`.
В микросервисной архитектуре один клик пользователя может породить 15 HTTP-запросов между 8 разными сервисами, запущенными в 30 случайных контейнерах. Искать ошибку руками — это безумие.

Системе нужны "Три столпа наблюдаемости" (Three Pillars of Observability).

#### 1. Logging (Логирование) — ELK Stack

Логи — это детальная запись произошедших событий (текст).

* **Проблема:** Невозможно зайти по SSH на 50 серверов и искать ошибку утилитой `grep`.
* **Решение (Централизация):** Мы используем паттерн **Log Aggregation**.
* Все микросервисы пишут логи не просто текстом, а в формате **JSON** (Structured Logging), чтобы их было легко парсить.
* Специальный агент (например, Logstash или Fluentd, работающий в K8s как DaemonSet) собирает эти файлы и отправляет в хранилище.
* Вся эта масса логов хранится в **Elasticsearch** (мощном поисковике на обратных индексах).
* Инженер открывает **Kibana** (веб-интерфейс) и за секунду находит логи со всех 50 серверов по запросу `user_id: "123" AND level: "ERROR"`.



#### 2. Metrics (Метрики) — Prometheus и Grafana

Метрики — это числа (Time-Series Data). Логи дорогие, они занимают терабайты. Метрики дешевые, это просто счетчики.
Мы не пишем в лог "Процессор загружен на 80%". Мы используем метрики.

* **Prometheus:** Индустриальный стандарт для сбора метрик. У него архитектура **Pull** (он сам раз в 10 секунд ходит по всем твоим микросервисам по URL `/actuator/prometheus` и забирает числа).
* *Типы метрик:* **Counter** (счетчик, только растет: количество заказов), **Gauge** (может расти и падать: текущее использование RAM), **Histogram** (распределение: 95-й перцентиль времени ответа БД).


* **Grafana:** Рисует из этих унылых цифр красивые графики. И самое главное — позволяет настроить **Алерты** (если `error_rate > 5%`, отправить сообщение дежурному в Telegram или позвонить ему ночью через PagerDuty).

#### 3. Distributed Tracing (Распределенная трассировка) — Jaeger

Это самый важный инструмент для System Design. Как понять, в каком именно из 8 сервисов завис запрос пользователя?

**Как это работает:**

1. Пользователь нажимает "Купить". Запрос приходит на API Gateway.
2. Gateway генерирует уникальный **Trace ID** (например, `abc-123`) и кладет его в HTTP-заголовок `X-B3-TraceId`.
3. API Gateway отправляет этот ID в сервис Заказов. Сервис Заказов, делая запрос в сервис Платежей, **обязан** передать этот же `Trace ID` дальше. И так по всей цепочке.
4. Каждый сервис, выполнив свою работу (это называется **Span**), асинхронно отправляет в систему трассировки (например, **Jaeger** или Zipkin) отчет: "Я часть Trace ID `abc-123`, моя работа (Span) заняла 45 миллисекунд".
5. Инженер открывает Jaeger, вбивает `Trace ID` и видит **Гант-диаграмму** всего запроса. Он визуально видит: "Ага, общий запрос занял 2 секунды. Из них 1.8 секунды мы ждали ответа от стороннего банковского API в сервисе Платежей. Вот где узкое место!".

**Итог Главы 18:**
Чтобы управлять распределенной системой вслепую, нам нужны:

* **Трассировка (Jaeger)** — чтобы увидеть путь запроса сквозь все сервисы (Trace ID) и найти узкое место.
* **Метрики (Prometheus/Grafana)** — чтобы видеть графики здоровья системы (CPU, тайминги, количество ошибок) и получать алерты ночью.
* **Централизованные Логи (ELK)** — чтобы, получив алерт по метрикам и найдя больной сервис по трассировке, вбить туда ID и прочитать точную причину падения (NullPointerException) в конкретной строке кода.
