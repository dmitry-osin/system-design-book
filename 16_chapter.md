**Глава 16. Паттерны работы с данными в микросервисах**

Это самая сложная часть микросервисов. Когда база данных была одна (Монолит), мы просто использовали транзакции (ACID) — `BEGIN`, `COMMIT`, `ROLLBACK`. В микросервисах базы данных разделены сетью. Как обновить данные в трех сервисах одновременно и не получить рассинхрон?

#### 1. CQRS (Command Query Responsibility Segregation)

**Проблема:** В монолите мы читаем и пишем в одну БД. В микросервисах мы разбили данные. Сервису "Личный кабинет" нужно показать пользователю список его заказов и статус их оплаты. Но заказы лежат в `OrderDB`, а оплаты в `PaymentDB`. Делать JOIN по сети — убийственно медленно.

**Решение (CQRS):** Мы физически разделяем модели Записи (Command) и Чтения (Query).

* **Command (Запись):** Клиент отправляет запрос "Оплатить заказ". Он идет в сложную бизнес-логику, которая валидирует правила и сохраняет данные в "Пишущую БД".
* Сразу после этого система кидает событие (Event) в Kafka: "Заказ 123 оплачен".
* **Query (Чтение):** Специальный сервис "Личный кабинет" слушает это событие из Kafka и обновляет свою собственную, отдельную "Читающую БД" (например, Elasticsearch или Mongo), которая заранее оптимизирована (денормализована) для мгновенного отображения на фронтенде. Клиент читает данные отсюда.

#### 2. Event Sourcing (Порождение событий)

Тесно связан с CQRS. Вместо того чтобы хранить текущее *состояние* (Баланс = 100$), мы храним **только историю событий**, которые привели к этому состоянию.

* *Хранилище:* Баланс 0$. `Пополнение +50$`, `Перевод -10$`, `Пополнение +60$`.
* *Как узнать баланс:* Мы просто суммируем все события при запросе (или держим закэшированный snapshot).
* *Зачем это нужно:* 100% Audit Log (идеально для финтеха, история неизменяема). Возможность "путешествий во времени" — можно восстановить состояние системы на любую секунду в прошлом, просто "проиграв" события до нужной точки.

#### 3. Распределенные транзакции (Saga Pattern)

Как гарантировать транзакцию, если сервиса два?
Двухфазный коммит (2PC) мы отметаем — он блокирует обе базы и не масштабируется. Используем **Saga (Сага)**.

Сага — это цепочка локальных транзакций. Каждый сервис делает свою работу и публикует событие в брокер (Kafka/RabbitMQ), запуская следующий шаг.

Если на шаге 3 происходит ошибка (на складе нет товара), Сага запускает **Компенсирующие транзакции** в обратном порядке:

1. *OrderService:* Создал заказ "В ожидании" -> кинул эвент.
2. *PaymentService:* Списал деньги -> кинул эвент.
3. *InventoryService:* ОШИБКА! Товара нет. -> кидает эвент ошибки.
4. *(Компенсация)* *PaymentService:* Возвращает деньги пользователю -> кидает эвент.
5. *(Компенсация)* *OrderService:* Ставит заказу статус "Отменен".

Есть два вида управления Сагой:

* **Choreography (Хореография):** Сервисы просто кидают события друг другу, как танцоры на сцене. Нет центрального дирижера. (Хорошо для 2-3 шагов).
* **Orchestration (Оркестрация):** Есть выделенный микросервис "Оркестратор" (например, на базе Camunda). Он как кукловод говорит каждому сервису: "Сделай это. Теперь ты сделай то. Ошибка? Возвращай деньги". (Обязательно для сложных процессов).

#### 4. Transactional Outbox (Святой Грааль микросервисов)

**Самая частая ошибка новичка (Dual Write Problem):**

```java
// Плохой код!
public void createOrder() {
    db.save(order); // Сохранили в Postgres
    kafka.send("OrderCreated", order); // Отправили в Kafka для других сервисов
}

```

*Что тут не так?* Если после `db.save()` сеть моргнет и Kafka недоступна, мы сохраним заказ в БД, но никто в системе об этом не узнает! Базы разъедутся.

**Паттерн Transactional Outbox (Исходящее письмо):**
Мы должны сделать сохранение в БД и отправку в Kafka атомарными.

1. Мы создаем в нашем Postgres вторую таблицу — `Outbox` (исходящие).
2. В рамках **одной локальной транзакции БД** мы сохраняем Заказ в таблицу `orders` И записываем событие в таблицу `outbox`. (Если сеть падает, откатываются обе записи — база сама гарантирует ACID).
3. Отдельный фоновый процесс (например, Debezium или Spring Scheduler) постоянно читает таблицу `outbox` и гарантированно, асинхронно перекладывает эти события в Kafka.

**Пример на Java (Outbox Pattern):**

```java
import org.springframework.transaction.annotation.Transactional;
import org.springframework.stereotype.Service;

@Service
public class OrderService {

    private final OrderRepository orderRepo;
    private final OutboxEventRepository outboxRepo;

    @Transactional // Критически важно! Обе вставки идут в одной транзакции
    public void createOrder(OrderRequest request) {
        // 1. Сохраняем бизнес-сущность
        Order order = new Order(request.getUserId(), request.getItem());
        orderRepo.save(order);

        // 2. В этой же транзакции создаем событие для Kafka в таблице БД!
        String eventPayload = "{ \"orderId\": " + order.getId() + ", \"status\": \"CREATED\" }";
        OutboxEvent event = new OutboxEvent(
                "order_topic", // куда отправить
                order.getId().toString(), // ключ маршрутизации
                eventPayload
        );
        outboxRepo.save(event);
        
        // Транзакция коммитится в БД целиком. Данные 100% не разъедутся.
    }
}

```

**Итог Главы 16:**

* В микросервисах данные размазаны по сети.
* **CQRS** ускоряет чтение, разделяя модели записи и чтения.
* **Event Sourcing** хранит данные как бесконечный лог банковских операций.
* **Saga** заменяет ACID-транзакции, управляя цепочкой шагов и откатами (компенсациями).
* **Transactional Outbox** решает фатальную проблему отправки сообщения в брокер после коммита в базу, делая это через промежуточную таблицу в БД.
