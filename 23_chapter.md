**Глава 23. Специфика финансовых и транзакционных систем (Fintech & Ledgers)**

#### 1. Двойная запись (Double-entry bookkeeping)

Это принцип, придуманный итальянскими купцами еще в 15 веке, и до сих пор это основа любого процессинга, криптобиржи или банка.

**Проблема:** В обычной системе (например, в игре) мы можем просто сделать `UPDATE users SET coins = coins + 100`. Деньги берутся "из воздуха". В реальном мире деньги не могут появиться из ниоткуда и не могут исчезнуть. Они могут только **перемещаться**.

**Решение:** Каждая финансовая операция — это всегда **две** (или более) симметричные записи в базе данных (Леджер / Главная книга). Сумма всех изменений в рамках одной транзакции всегда должна быть равна нулю.

*Пример:* Дмитрий переводит 1000 рублей Ивану.
Мы не обновляем два баланса. Мы делаем две записи в таблицу `Ledger`:

1. `Счет: Дмитрий, Сумма: -1000 руб, Валюта: RUB, Тип: Перевод`
2. `Счет: Иван, Сумма: +1000 руб, Валюта: RUB, Тип: Пополнение`

*Зачем это нужно:*

* **Баланс системы:** Если ночью запустить скрипт `SELECT SUM(Сумма) FROM Ledger`, результат должен быть строго равен `0`. Если там `+50` или `-10`, значит в коде баг, и система либо "напечатала" деньги, либо уничтожила их.
* Если мы берем комиссию за перевод (50 рублей), транзакция будет состоять из трех записей: Дмитрий (-1050), Иван (+1000), Счет_Комиссий_Банка (+50). Сумма снова равна нулю!

#### 2. Строгая идемпотентность (Idempotency Keys)

Мы обсуждали идемпотентность очередей в Главе 7, но в финтехе она начинается еще на уровне HTTP API (до очередей).

**Проблема:** Пользователь нажимает кнопку "Оплатить 5000 руб". Мобильный интернет зависает. Телефон пользователя не получает ответ от сервера и через 10 секунд делает *автоматический повтор* (Retry) запроса.
Если бэкенд тупой, он обработает оба запроса, и с карты спишется 10 000 руб.

**Решение: Idempotency-Key (Ключ идемпотентности)**
Клиент (мобилка или чужой микросервис) **обязан** при каждом новом намерении сгенерировать уникальный ключ (обычно UUID) и передать его в HTTP-заголовке `Idempotency-Key`. При ретраях этот ключ не меняется!

*Как работает сервер:*

1. Получает запрос и проверяет ключ в Redis/БД.
2. Если ключа нет -> сохраняет ключ со статусом `IN_PROGRESS`, списывает деньги, меняет статус ключа на `SUCCESS` и сохраняет ответ (`{"status": "ok", "tx_id": 123}`).
3. Если ключ есть и статус `IN_PROGRESS` -> значит, первый запрос еще крутится. Мы возвращаем клиенту ошибку `409 Conflict` (Подождите, операция в обработке).
4. Если ключ есть и статус `SUCCESS` -> мы **не** идем списывать деньги! Мы просто достаем из кэша сохраненный ответ `{"status": "ok", "tx_id": 123}` и отдаем его клиенту. Клиент думает, что всё прошло успешно, а мы спасли его от двойного списания.

#### 3. Неизменяемость истории (Append-only) и Audit Logs

В финтехе операция `UPDATE` для финансовых записей **запрещена** на уровне архитектуры. Записи можно только добавлять в конец (Append-only).

Если оператор ошибся и перевел Ивану 5000 рублей вместо 500, он не может зайти в базу и поменять строку. Он должен сделать **Компенсирующую транзакцию**:

1. *(Ошибочная запись)* `Иван: +5000`
2. *(Компенсация)* `Иван: -5000 (Причина: Откат транзакции X)`
3. *(Правильная запись)* `Иван: +500`

Это называется Event Sourcing (Глава 16). Баланс пользователя не хранится статично, он вычисляется на лету (или кэшируется в отдельной таблице `balances` через CQRS) на основе суммы всех его неизменяемых исторических операций. Это дает 100% гарантию для любого аудита или налоговой проверки.

#### 4. Боль с плавающей точкой (The Float Problem)

Это классический вопрос на собеседованиях для Junior/Middle, но Senior должен закладывать это на уровне API контрактов.

**Проблема:** Типы данных `float` и `double` используют двоичную арифметику IEEE 754. Они не могут точно представить десятичные дроби.
Если в Java сложить `0.1 + 0.2`, результат будет `0.30000000000000004`. В масштабах миллионов транзакций эти копейки сложатся в огромные недостачи, и баланс (двойная запись) не сойдется!

**Решения:**

1. **Хранить в минимальных единицах (Integer/BigInt):** Рубли хранить в копейках, доллары — в центах, биткоины — в сатоши. `10.50$` хранится в БД как `1050`. Никаких дробей!
2. **Использовать точные типы (BigDecimal в Java / DECIMAL в SQL):** Они работают медленнее, но считают математически идеально.

**Пример на Java (Финтех-подход):**

```java
import java.math.BigDecimal;
import java.math.RoundingMode;

public class MoneyExample {
    public static void main(String[] args) {
        // ПЛОХО (Увольнение):
        double balance1 = 0.1;
        double balance2 = 0.2;
        System.out.println("Double: " + (balance1 + balance2)); 
        // Вывод: 0.30000000000000004

        // ХОРОШО (BigDecimal):
        // Важно: создаем BigDecimal ИЗ СТРОКИ, а не из double, 
        // иначе мы притащим неточность double внутрь BigDecimal
        BigDecimal exactBalance1 = new BigDecimal("0.1");
        BigDecimal exactBalance2 = new BigDecimal("0.2");
        
        BigDecimal total = exactBalance1.add(exactBalance2);
        System.out.println("BigDecimal: " + total); 
        // Вывод: 0.3
        
        // Пример деления (например, начисление 13% налога):
        // Обязательно указываем правило округления (RoundingMode),
        // иначе при бесконечной дроби (1/3) BigDecimal выбросит ArithmeticException
        BigDecimal tax = total.multiply(new BigDecimal("0.13"))
                              .setScale(2, RoundingMode.HALF_UP); // Округление до копеек
    }
}

```

**Итог Главы 23:**

* **Двойная запись:** Защищает систему от появления денег из ниоткуда. Сумма проводок внутри бизнес-транзакции всегда равна `0`.
* **Идемпотентные API:** Защищают пользователя от двойных списаний при лагах сети через уникальные заголовки `Idempotency-Key`.
* **Append-only:** Данные о транзакциях никогда не обновляются (`UPDATE` = зло), только компенсируются новыми записями (Event Sourcing).
* **Типы данных:** Никаких `float`/`double` в финансовых расчетах. Только целые числа (в центах) или `BigDecimal`.
