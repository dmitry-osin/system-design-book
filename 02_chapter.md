**Глава 2. Проектирование API (API Design)**

API (Application Programming Interface) — это язык, на котором общаются программы. От того, какой стиль API ты выберешь, будет зависеть скорость разработки, производительность системы и то, насколько удобно будет твоим клиентам (фронтенду, мобилкам или другим микросервисам).

#### 1. REST (Representational State Transfer)

Это самый популярный стандарт в индустрии. Он строится вокруг концепции **Ресурсов** (существительных) и стандартных **Методов HTTP** (глаголов).

* **Ресурсы:** Мы обращаемся к сущностям по URL. Например, `/users` или `/orders/123`.
* **Методы (CRUD):**
* `GET /users` — дай список пользователей (Чтение).
* `POST /users` — создай нового пользователя (Создание).
* `PUT /users/123` — полностью обнови пользователя 123 (Замещение).
* `PATCH /users/123` — частично обнови пользователя (например, только email).
* `DELETE /users/123` — удали пользователя.


* **Статус-коды:** REST обязывает правильно использовать коды ответов HTTP:
* `2xx` — Всё отлично (200 OK, 201 Created).
* `4xx` — Ошибка клиента (400 Bad Request, 401 Unauthorized, 404 Not Found).
* `5xx` — Ошибка сервера (500 Internal Server Error — мы где-то поймали NullPointerException, 503 Service Unavailable).



*Пример из жизни:* REST — это как поход в библиотеку со строгим каталогом. Ты даешь точный номер карточки (URL) и говоришь библиотекарю действие (Дай, Забери, Обнови).

#### 2. GraphQL

Разработан Facebook для решения проблем REST, особенно на мобильных устройствах. В REST, если тебе нужно показать страницу профиля с последними 3 постами, тебе часто нужно сделать 2-3 разных запроса (сначала `/users/123`, потом `/users/123/posts`). Это называется **Over-fetching** (тянем лишние данные) и **Under-fetching** (данных не хватает, нужен новый запрос).

В GraphQL есть только **одна точка входа** (обычно `POST /graphql`), и клиент сам пишет запрос, перечисляя ровно те поля, которые ему нужны.

*Проблема N+1 в GraphQL:*
Так как GraphQL резолвит графы, возникает классическая проблема баз данных. Допустим, мы запрашиваем список из 10 авторов и их книги.
Неопытный сервер сделает 1 запрос: "Дай 10 авторов" (1). А затем для *каждого* автора сделает запрос: "Дай книги для автора X" (N = 10). Итого 11 запросов в БД.
*Решение:* Использование паттерна **DataLoader**. Он собирает (батчит) все запросы к книгам в один пул и делает один большой SQL запрос: `SELECT * FROM books WHERE author_id IN (1, 2, ..., 10)`.

#### 3. RPC: gRPC и Protocol Buffers

Если REST и GraphQL обычно общаются в текстовом формате JSON (который медленно парсить и он занимает много места), то для общения *между микросервисами* (backend-to-backend) часто используют **gRPC** (от Google).

* **gRPC работает поверх HTTP/2:** А значит, из коробки имеет мультиплексирование (нет блокировок).
* **Protocol Buffers (Protobuf):** Это язык описания контрактов и бинарный формат сериализации. Данные сжимаются в бинарный код, который передается и парсится в разы быстрее, чем JSON.

**Пример на Java (gRPC & Protobuf):**

Сначала мы пишем контракт в файле `.proto` (независимо от языка программирования):

```protobuf
// Файл: payment.proto
syntax = "proto3";

// Описываем структуру запроса
message PaymentRequest {
  string user_id = 1;
  double amount = 2;
}

// Описываем структуру ответа
message PaymentResponse {
  bool success = 1;
  string transaction_id = 2;
}

// Описываем сам сервис и его методы (RPC)
service PaymentService {
  rpc ProcessPayment (PaymentRequest) returns (PaymentResponse);
}

```

Затем компилятор Protobuf генерирует для нас Java-классы. Наш код вызова выглядит так, как будто мы вызываем локальный метод:

```java
// Клиентский код в сервисе Корзины, который вызывает сервис Платежей
public class CartService {
    
    // Сгенерированный "stub" (заглушка), прячущая всю сетевую логику
    private final PaymentServiceGrpc.PaymentServiceBlockingStub paymentStub;

    public CartService(Channel channel) {
        this.paymentStub = PaymentServiceGrpc.newBlockingStub(channel);
    }

    public void checkout(String userId) {
        // 1. Создаем объект запроса через сгенерированный Builder
        PaymentRequest request = PaymentRequest.newBuilder()
                .setUserId(userId)
                .setAmount(99.99)
                .build();

        // 2. Делаем удаленный вызов (Remote Procedure Call). 
        // Под капотом: сериализация в бинарник -> HTTP/2 запрос -> ожидание -> десериализация.
        PaymentResponse response = paymentStub.processPayment(request);

        if (response.getSuccess()) {
            System.out.println("Оплата прошла: " + response.getTransactionId());
        }
    }
}

```

#### 4. Асинхронные API (Push вместо Pull)

Иногда клиенту нужно получать обновления в реальном времени. Постоянно дергать сервер (Long Polling) запросами `GET /status` — дорого и неэффективно.

* **WebSockets:** Двунаправленное, постоянное TCP-соединение. И клиент, и сервер могут кидать друг другу сообщения в любой момент.
* *Где применять:* Чаты (Telegram/WhatsApp), мультиплеер в браузере, real-time дашборды (биржевые графики).


* **Server-Sent Events (SSE):** Однонаправленный поток данных. Клиент подписывается по HTTP, соединение остается открытым, и сервер пушит туда события (текст). Клиент ответить в этот же канал не может.
* *Где применять:* Лента новостей, уведомления, трансляция логов сборки (как в Jenkins/GitLab CI), стриминг ответов от нейросетей (тот же ChatGPT или я возвращаем текст по словам через SSE).


* **Webhooks:** Общение Server-to-Server. Ты регистрируешь свой URL (например, `[https://my-shop.com/api/payment-callback](https://my-shop.com/api/payment-callback)`) в стороннем сервисе (Stripe/ЮKassa). Когда происходит событие (деньги списались), чужой сервер делает `POST` запрос на твой URL.
* *Где применять:* Интеграция с платежными шлюзами, боты для Telegram, GitHub Actions.



---

**Итог:**

* **REST** — универсальный стандарт, основанный на HTTP глаголах и ресурсах. Отлично подходит для публичных API и взаимодействия Client-Server.
* **GraphQL** — гибкий язык запросов, позволяющий фронтенду забрать все нужные данные одним запросом, решая проблемы over/under-fetching, но требующий внимания к проблеме N+1 на бэкенде.
* **gRPC (Protobuf)** — бинарный, сверхбыстрый стандарт на базе HTTP/2. Идеален для внутренних коммуникаций между микросервисами, где критична скорость и строгие контракты.
* **WebSockets, SSE, Webhooks** — инструменты для событийной (event-driven) архитектуры, когда нужно передавать данные в реальном времени без постоянного опроса сервера (polling).
