**Глава 13. Алгоритмы консенсуса и время**

#### 1. Выборы лидера и проблема Split-Brain

В архитектуре Master-Slave (которую мы обсуждали в Главе 10) все записи идут в Master-узел. Но что если Master внезапно сгорает?
Система должна автоматически выбрать нового лидера из оставшихся Slave-узлов. Это называется **Leader Election (Выборы лидера)**.

**Смертельная проблема — Split-Brain (Расщепление мозга):**
Представь 5 серверов. 2 в Европе, 3 в США. Пропадает трансатлантический кабель. Европа думает: "США умерли, надо выбрать лидера среди нас". США думают: "Европа умерла, выбираем лидера у нас".
В итоге в системе появляется **ДВА** Master-узла. Клиенты начинают писать разные данные в оба. Когда кабель починят, базы данных будут безнадежно испорчены несовместимыми конфликтами.

**Решение — Строгое Большинство (Кворум Консенсуса):**
Чтобы выбрать лидера, кандидат должен набрать большинство голосов: `(N / 2) + 1`.

* Если у нас 5 серверов, большинство — это 3.
* При обрыве кабеля Европа (2 узла) не сможет набрать 3 голоса. Они перейдут в режим "Только чтение" (пожертвовав Availability ради Consistency). США (3 узла) наберут большинство, выберут лидера и продолжат работу. Split-Brain невозможен!
* *Именно поэтому кластеры баз данных (ZooKeeper, Kafka, etcd) всегда разворачивают нечетным числом: 3, 5 или 7 серверов.*

#### 2. Алгоритмы Консенсуса (Paxos, Raft, Zab)

Как именно узлы договариваются и голосуют?

* **Paxos:** Исторически первый алгоритм консенсуса. Он настолько математически сложен, что мало кто из инженеров понимает его до конца.
* **Zab (ZooKeeper Atomic Broadcast):** Разработан специально для Apache ZooKeeper (менеджера конфигураций, который используют Kafka и Hadoop).
* **Raft:** Самый популярный современный алгоритм (используется в etcd, Consul, MongoDB). Он был создан специально, чтобы быть понятным для человека.

**Как работает Raft (на пальцах):**

1. Все узлы запускаются как *Followers* (Слушатели). У каждого есть случайный таймер (например, от 150 до 300 мс).
2. Если таймер истекает, а от Лидера нет вестей, узел объявляет себя *Candidate* (Кандидатом), голосует сам за себя и рассылает запрос остальным: "Голосуйте за меня!".
3. Кто первый набрал большинство — становится *Leader* (Лидером).
4. Лидер начинает постоянно (каждые 50 мс) рассылать всем *Heartbeats* (Пульс) — пустые сообщения, которые сбрасывают таймеры Слушателей.
5. Если Лидер умирает, пульс пропадает, таймеры истекают, и выборы начинаются заново.

#### 3. Gossip Protocols (Распространение слухов)

Консенсус работает отлично для маленьких кластеров (5-7 узлов). Но что если у нас кластер Cassandra из 1000 серверов? Запускать Raft на 1000 серверов нельзя — сеть ляжет от количества сообщений для голосования.

В огромных системах используют **Эпидемические протоколы (Gossip)**.

* *Аналогия:* Как сплетни распространяются в офисе? Алиса рассказывает новость Бобу и Чарли. На следующей минуте Боб рассказывает Дейву и Еве, а Чарли — Фрэнку и Грейс. Новость распространяется в геометрической прогрессии (экспоненциально).
* В системе каждый сервер раз в секунду выбирает 2-3 случайных других сервера и обменивается с ними информацией: "Сервер №54 упал, а на Сервере №102 место на диске заканчивается".
* *Результат:* За несколько секунд без всякого центрального координатора все 1000 узлов в кластере знают точное состояние системы.

#### 4. Проблема Времени: Почему нельзя верить системным часам?

Представь, что Сервер А записывает значение `X=1` в 12:00:01. Сервер Б записывает `X=2` в 12:00:02. Какое значение верное? Кажется, что второе.

**Но в распределенных системах `System.currentTimeMillis()` — это ложь.**
Кварцевые генераторы на материнских платах неидеальны. Часы на серверах постоянно спешат или отстают (Clock Drift). Даже с синхронизацией через интернет (NTP) разница между двумя серверами может составлять десятки миллисекунд. В HighLoad за это время происходят тысячи транзакций. Сервер Б мог записать `X=2` *раньше*, просто его часы отставали!

**Решение 1: Lamport Timestamps (Логические часы Лампорта)**
Мы отказываемся от физического времени. Вместо этого каждое сообщение в системе получает логический счетчик (целое число).

*Правило Лампорта:*

1. Каждый сервер имеет свой счетчик (сначала 0).
2. Перед выполнением любого действия сервер делает `counter++`.
3. Отправляя сообщение другому серверу, он прикрепляет свой `counter`.
4. Принимающий сервер смотрит на счетчик из сообщения. Если он больше его собственного, сервер обновляет свой счетчик: `counter = max(local_counter, message_counter) + 1`.

**Решение 2: Vector Clocks (Векторные часы)**
Лампорт показывает последовательность, но не видит *конфликты* параллельных независимых действий. Векторные часы решают это. Это массив счетчиков для каждого сервера (вспоминаем CRDT из прошлой главы).
Например: `[Лондон: 2, Токио: 1, Нью-Йорк: 0]`.
Если при слиянии данных Токио видит вектор `[Лондон: 2, Токио: 1]` и вектор `[Лондон: 1, Токио: 5]`, система математически понимает, что это конфликт параллельных записей, который должен разрешить программист или клиентское приложение (так работает база данных Amazon Dynamo).

**Пример на Java (Логические часы Лампорта):**

```java
import java.util.concurrent.atomic.AtomicInteger;

public class LamportClock {
    private final AtomicInteger counter = new AtomicInteger(0);

    // Событие происходит внутри самого сервера (например, сохранили в БД)
    public int tick() {
        return counter.incrementAndGet();
    }

    // Отправляем сообщение по сети: счетчик тикает + мы отдаем его значение
    public int send() {
        return counter.incrementAndGet();
    }

    // Получаем сообщение из сети с чужим счетчиком
    public int receive(int receivedTimestamp) {
        // Берем максимальное из нашего текущего времени и времени из сообщения,
        // а затем увеличиваем на 1.
        return counter.updateAndGet(current -> Math.max(current, receivedTimestamp) + 1);
    }
}

```

---

**Итог:**

* **Сплит-брэйн (Split-Brain)** — смертельная угроза потери консистентности при разделении сети. Решается введением нечетного количества серверов и правилом "Строгого большинства" для принятия решений.
* **Raft и Paxos** — алгоритмы консенсуса, позволяющие узлам децентрализованно выбрать Лидера и договориться о состоянии системы.
* **Gossip Protocols** — заменяют Консенсус в огромных кластерах (сотни узлов), позволяя серверам узнавать о сбоях соседей через эпидемическое "распространение слухов".
* **Физическое время лжет.** В распределенных системах мы используем логическое время (Логические часы Лампорта или Векторные часы), чтобы отслеживать причинно-следственные связи и выявлять конфликты.
