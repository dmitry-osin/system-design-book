**Глава 22. Безопасность (Security by Design)**

Для начала — золотое правило: никогда не придумывай свои алгоритмы криптографии. Используй проверенные индустриальные стандарты.

#### 1. База: Аутентификация vs Авторизация

Частая ошибка на собеседованиях — путать эти два понятия.

* **Authentication (Аутентификация - AuthN):** Проверка того, *кто* ты такой. (Пользователь вводит логин и пароль. Система говорит: "Да, ты Дмитрий").
* **Authorization (Авторизация - AuthZ):** Проверка того, *что* тебе разрешено делать. (Дмитрий просит удалить базу данных. Система говорит: "У тебя роль 'Developer', а не 'Admin'. Отказано").

#### 2. Session-based Auth vs JWT

Как система "запоминает" пользователя после того, как он ввел пароль?

**Session-based (Stateful):**
Исторический подход монолитов. Пользователь логинится. Сервер создает объект `Session` в оперативной памяти (или в Redis), генерирует случайный `session_id` (например, `abc123`) и отдает его браузеру в Cookie.
С каждым новым запросом браузер присылает этот `session_id`, а сервер ищет его в Redis, чтобы понять, кто это.

* *Плюсы:* Идеальный контроль. Если у пользователя украли ноутбук, админ просто удаляет `session_id` из Redis, и пользователя мгновенно "выкидывает" из системы.
* *Минусы:* Тяжело масштабировать. Все микросервисы должны постоянно ходить в Redis, чтобы проверять сессии.

**JWT - JSON Web Token (Stateless):**
Стандарт распределенных систем. Сервер ничего не сохраняет у себя!
После логина сервер генерирует длинную строку (Токен), состоящую из трех частей, разделенных точками: `Header.Payload.Signature`.

* *Header:* Тип токена и алгоритм (например, `HS256`).
* *Payload (Полезная нагрузка):* JSON с данными: `{"user_id": 42, "role": "user", "exp": 1700000000}`.
* *Signature (Подпись):* Самая важная часть. Сервер берет Header, Payload и свой **секретный ключ** (который знает только он), и прогоняет через криптографическую хэш-функцию.

*Как это работает:* Клиент присылает JWT. Сервер берет Payload из токена, снова подписывает его своим секретным ключом и проверяет: "Совпадает ли моя новая подпись с той, что прислал клиент?". Если совпадает — токен подлинный, никто не подделал данные. Сервер сразу знает, что `user_id = 42`. Никаких запросов в базу!

**Огромная проблема JWT:** Его невозможно отозвать (Invalidation). Если хакер украл JWT, он будет им пользоваться, пока не истечет время `exp` (Expiration time).

* *Решение (Паттерн Access / Refresh Token):*
1. Мы выдаем короткоживущий **Access Token** (JWT на 15 минут).
2. И долгоживущий **Refresh Token** (Случайная строка на 30 дней, которая хранится в БД сервера).
3. Когда Access Token протухает через 15 минут, клиент отправляет Refresh Token на сервер авторизации. Сервер проверяет БД (не забанили ли юзера?) и выдает новый JWT на 15 минут. Если украли ноутбук, мы удаляем Refresh Token из БД, и хакер потеряет доступ максимум через 15 минут.



#### 3. OAuth 2.0 и OpenID Connect (OIDC)

Когда ты видишь кнопку *"Войти через Google"*, под капотом работает OAuth 2.0.

* **OAuth 2.0:** Это протокол **Авторизации** (делегирования прав).
* *Аналогия:* Ты нанимаешь уборщика в квартиру. Ты не даешь ему свой главный ключ (пароль), который открывает еще и сейф. Ты даешь ему "временный ключ", который открывает только входную дверь и работает только с 12:00 до 15:00.
* *Пример:* Твое приложение просит у Google доступ к календарю пользователя. Google спрашивает пользователя: "Разрешить?". Если да, Google выдает твоему приложению токен доступа (Access Token), который годится *только* для чтения календаря.


* **OpenID Connect (OIDC):** Это надстройка над OAuth 2.0 для **Аутентификации**. OAuth выдает "ключ от двери", а OIDC выдает "паспорт" (ID Token, обычно в формате JWT), в котором написано имя пользователя, его email и аватарка.

#### 4. Шифрование и mTLS (Mutual TLS)

Данные нужно защищать в двух состояниях.

**1. Data at Rest (Данные в покое):**
Если кто-то вынесет жесткий диск из дата-центра, он не должен прочитать данные. Базы данных (Postgres, S3) настраиваются на автоматическое шифрование дисков (Transparent Data Encryption - TDE) симметричными ключами (AES-256). Пароли пользователей **никогда** не хранятся в открытом виде, только их криптографические хэши (например, через алгоритм `bcrypt` или `Argon2`, которые специально работают медленно, чтобы защитить от подбора / Brute-force).

**2. Data in Transit (Данные в пути) и mTLS:**
Пользователь обращается к нашему API Gateway через защищенный HTTPS (TLS). Трафик зашифрован.
А что происходит за Gateway, во внутренней сети Kubernetes? Раньше считалось, что внутренняя сеть безопасна, и микросервисы общались по простому HTTP.
*В Zero Trust это недопустимо.* Если хакер взломает один слабый микросервис, он сможет "слушать" весь трафик между остальными сервисами внутри K8s.

*Решение — mTLS (Взаимный TLS):*
В обычном интернете (HTTPS) клиент проверяет паспорт (сертификат) сервера.
В mTLS **оба** участника проверяют паспорта друг друга.

1. Сервис Заказов стучится в Сервис Платежей.
2. Сервис Платежей говорит: "Покажи свой сертификат".
3. Сервис Заказов показывает. Сервис Платежей проверяет, выдан ли он нашим внутренним Удостоверяющим Центром.
4. Затем Сервис Платежей показывает свой сертификат Сервису Заказов.
5. Только после обоюдной проверки они устанавливают зашифрованный туннель.
*(Обычно это настраивается не в коде Java, а на уровне инфраструктуры через Service Mesh, например, Istio или Linkerd).*

**Пример на Java (Почему bcrypt так важен):**

```java
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;

public class SecurityExample {
    public static void main(String[] args) {
        // Мы используем bcrypt с "солью" (salt). 
        // Алгоритм сам генерирует случайную соль и вшивает ее в результат.
        BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
        
        String myPassword = "super_secret_password";
        
        String hash1 = encoder.encode(myPassword);
        String hash2 = encoder.encode(myPassword);
        
        // Хэши будут РАЗНЫМИ из-за случайной соли! 
        // Это защищает базу данных от атаки по "радужным таблицам" (Rainbow Tables),
        // даже если у двух пользователей одинаковые пароли.
        System.out.println("Hash 1: " + hash1);
        System.out.println("Hash 2: " + hash2);
        
        // Spring сам достанет соль из хэша и проверит пароль
        boolean isMatch = encoder.matches("super_secret_password", hash1);
        System.out.println("Пароль верный? " + isMatch);
    }
}

```

---

**Итог Главы 22:**

* Разделяй **Аутентификацию** (проверка личности) и **Авторизацию** (проверка прав).
* В микросервисах используй **Stateless JWT токены**, но обязательно комбинируй короткоживущие Access токены с отзываемыми Refresh токенами.
* **OAuth 2.0** — стандарт делегирования доступов, **OIDC** — стандарт аутентификации через внешних провайдеров (Google, Apple).
* Внутренняя сеть не безопасна (Zero Trust). Все межсервисные коммуникации должны быть зашифрованы и аутентифицированы через **mTLS**. Пароли в БД всегда хэшируем через `bcrypt/Argon2`.
