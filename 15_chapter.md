### Уровень 4: Продвинутые паттерны и эксплуатация (Advanced & Resiliency)

*Как строить сложные системы и безопасно их развивать.*

**Глава 15. Микросервисная архитектура (Microservices)**

Главное правило микросервисов: **Это не технологическое решение, это организационное решение**. Микросервисы нужны не для того, чтобы приложение работало быстрее (оно будет работать медленнее из-за сети), они нужны, чтобы *команды программистов* работали быстрее и независимо друг от друга.

#### 1. Decomposition: Как делить монолит (DDD)

Самая частая ошибка новичков — попилить монолит по техническим слоям (сервис базы данных, сервис бизнес-логики, сервис API) или напилить 500 "нано-сервисов" по одной функции в каждом. В итоге получается "Распределенный монолит", который падает целиком при любой ошибке.

Правильный подход — **Domain-Driven Design (DDD - Предметно-ориентированное проектирование)**.
Мы делим систему по бизнес-смыслу (Доменам).

* **Bounded Context (Ограниченный контекст):** У каждого домена свои строгие границы и своя терминология.
* *Пример:* Слово "Клиент" для сервиса Доставки — это адрес и номер телефона. Для сервиса Биллинга "Клиент" — это номер кредитной карты и баланс. Это должны быть два совершенно разных микросервиса (Доставка и Биллинг), каждый со своей независимой базой данных. Они не имеют права лезть в чужую БД напрямую!



#### 2. Service Discovery (Обнаружение сервисов)

Вспоминаем Kubernetes (Глава 5). Поды постоянно умирают и рождаются, их IP-адреса меняются каждую минуту.

Как сервису Корзины отправить HTTP-запрос в сервис Платежей, если он не знает его IP?

* В Kubernetes это решено "из коробки" через внутренний DNS (CoreDNS) и сущность `Service`. Ты просто делаешь запрос на `http://payment-service`, а K8s сам находит живой IP.
* Если ты не в K8s (например, на голых виртуалках AWS), тебе нужен **Service Registry** (Consul, Netflix Eureka). Это как адресная книга.
1. Сервер Платежей запускается и кричит в Consul: "Я тут, мой IP 10.0.0.5".
2. Сервис Корзины спрашивает Consul: "Где мне найти Платежи?". Consul отдает список живых IP.



#### 3. API Gateway и BFF

Представь, что у нас 20 микросервисов. Мобильное приложение пользователя (Фронтенд) не должно само обращаться к каждому из них (вспоминаем проблему N+1 запросов и задержки мобильного интернета).

* **API Gateway (Шлюз API):** Это единая точка входа в систему (например, Nginx, Kong, Spring Cloud Gateway). Все клиенты стучатся только в него.
* *Что он делает:* **Роутинг** (перенаправляет `/api/orders` в сервис заказов), **Аутентификация** (проверяет JWT-токен пользователя один раз, чтобы 20 микросервисам не приходилось делать это самим), **Rate Limiting** (защита от DDoS — не больше 100 запросов в секунду от одного IP).


* **BFF (Backend for Frontend):** Это развитие идеи API Gateway. У мобильного приложения (узкий экран) и десктопного сайта (широкий экран) разные потребности. Десктопу нужно 50 полей данных, мобилке — 10. BFF — это специальный микросервис-прослойка, который собирает данные из 5 разных внутренних сервисов, агрегирует их, обрезает лишнее и отдает мобилке идеальный, легкий JSON за один запрос. У мобилки свой BFF, у веб-сайта — свой.

**Итог Главы 15:**

* Пилить монолит нужно строго по бизнес-доменам (DDD), давая каждому сервису свою изолированную БД.
* Микросервисы находят друг друга в динамическом облаке через Service Discovery.
* API Gateway прячет зоопарк микросервисов от пользователя, забирая на себя безопасность и роутинг, а BFF оптимизирует данные под конкретные устройства (iOS, Web).
