### Уровень 1: Фундамент (The Basics)

**Глава 1. Основы сетевого взаимодействия**

#### 1. Модель OSI и TCP/IP: L4 Transport и L7 Application

Исторически придумали 7-уровневую модель OSI, чтобы стандартизировать сети. В реальности интернет работает на более простой 4-уровневой модели TCP/IP. Как инженеру, проектирующему системы, тебе нужно кристально понимать разницу между двумя уровнями: **L4 (Транспортный)** и **L7 (Прикладной)**.

* **L4 (Transport):** Отвечает за доставку байтов от порта сервера А к порту сервера Б. Ему абсолютно всё равно, *что* внутри этих байтов — JSON, картинка или зашифрованный мусор. Главные игроки здесь: TCP и UDP.
* **L7 (Application):** Отвечает за смысл данных. Это протоколы, которые понимают наши приложения: HTTP, WebSocket, gRPC, SMTP (почта). Они строятся поверх L4.

**Пример из жизни:**
Представь курьерскую службу.
**L4** — это грузовик и правила дорожного движения. Задача — довезти коробку по адресу.
**L7** — это то, как ты упаковал подарок внутри коробки и написал инструкцию "Хрупкое, открывать осторожно". Курьеру (L4) плевать на подарок, он везет коробку. Получателю (L7) важен сам подарок и инструкция.

#### 2. TCP vs UDP: Гарантии vs Скорость

На транспортном уровне (L4) у нас есть два главных инструмента.

**TCP (Transmission Control Protocol)**
Это надежный протокол с установкой соединения (Handshake). Он гарантирует:

1. **Доставку:** Если пакет потерялся, TCP запросит его снова.
2. **Порядок:** Пакеты соберутся строго в том порядке, в котором были отправлены.
3. **Контроль перегрузки:** Если сеть «задыхается», TCP снизит скорость отправки.

Где используется: REST API, базы данных (PostgreSQL, Redis), финансовые транзакции. Всё, где потеря байта — это критическая ошибка.

**UDP (User Datagram Protocol)**
Это протокол "отправил и забыл". Никаких гарантий доставки, никакого порядка. Пакет может потеряться, продублироваться или прийти раньше предыдущего. Зато он работает **максимально быстро**, так как не тратит время на проверки и рукопожатия.

Где используется: Видеоконференции (Zoom), стриминг, и особенно — динамичные многопользовательские игры.
*Пример:* В шутерах вроде CS2 или Quake координаты игроков (где прицел, куда бежит) отправляются по UDP десятки раз в секунду. Если пакет с координатами потеряется, серверу нет смысла просить его заново — через миллисекунду прилетит новый пакет с *актуальными* координатами. А вот чат в той же игре или авторизация будут работать по TCP.

**Пример на Java (Разница в API):**
Посмотри, как эта разница отражается в стандартной библиотеке Java. TCP работает с *потоками* (Stream), а UDP с независимыми *пакетами* (Datagram).

```java
import java.net.*;
import java.io.*;

public class NetworkExamples {

    // TCP: Работаем как с непрерывной трубой (Stream)
    public void tcpExample() throws Exception {
        // Устанавливаем надежное соединение (Handshake)
        try (Socket socket = new Socket("google.com", 80);
             OutputStream out = socket.getOutputStream()) {
             
            // Мы просто пишем в поток, TCP сам разобьет это на пакеты 
            // и гарантирует, что они дойдут по порядку.
            out.write("GET / HTTP/1.1\r\n\r\n".getBytes());
        }
    }

    // UDP: Кидаем независимые коробки (Datagrams)
    public void udpExample() throws Exception {
        try (DatagramSocket socket = new DatagramSocket()) {
            byte[] message = "Ping".getBytes();
            InetAddress address = InetAddress.getByName("192.168.1.100");
            
            // Формируем конкретный пакет. Если он потеряется в сети, 
            // Java (и ОС) ничего не сделают. Он просто исчезнет.
            DatagramPacket packet = new DatagramPacket(message, message.length, address, 8080);
            socket.send(packet);
        }
    }
}

```

#### 3. DNS (Domain Name System)

Интернет работает на IP-адресах (например, `142.250.190.46`), но люди их не запоминают. DNS — это распределенная телефонная книга, переводящая `google.com` в IP-адрес.

**Как работает резолвинг (поиск):**

1. **Кэш браузера/ОС:** Сначала твой компьютер проверяет, не помнит ли он этот адрес.
2. **ISP Resolver:** Если нет, запрос идет к провайдеру (или к публичным DNS вроде `8.8.8.8` от Google).
3. **Root & TLD Servers:** Если провайдер не знает, он идет к корневым серверам, те отправляют его к серверам зоны (например, `.com`), а те уже к конкретному DNS-серверу домена.

*Для System Design:* DNS часто используется для простейшей балансировки нагрузки. Один домен может резолвиться в разные IP-адреса (DNS Round Robin) или отдавать ближайший к пользователю IP (GeoDNS).

#### 4. Эволюция HTTP (L7 протокол)

Вся веб-разработка стоит на HTTP. Понимание его версий критично для оптимизации производительности.

* **HTTP/1.1:** Текстовый протокол. Главная проблема — *Head-of-Line (HoL) blocking на уровне HTTP*. В одном TCP-соединении запросы выполняются строго по очереди. Если сервер долго генерирует первую картинку, остальные файлы ждут. Приходилось открывать по 6 параллельных TCP-соединений к серверу.
* **HTTP/2:** Бинарный протокол. Ввел **Мультиплексирование**. Теперь в рамках *одного* TCP-соединения можно параллельно отправлять и получать кучу разных файлов. Это решило HoL на уровне HTTP. *Но остался HoL на уровне TCP!* Если в сети потеряется хоть один TCP-пакет, весь поток данных останавливается и ждет его переотправки, даже если потерянный пакет относился к картинке, а мы ждем CSS.
* **HTTP/3:** Революция. Работает поверх **QUIC** (который, в свою очередь, работает поверх **UDP**!). Протокол сам реализует надежность, но делает это независимо для каждого потока данных. Потерялся пакет от картинки? Картинка подождет, но CSS загрузится без задержек. Идеально для нестабильного мобильного интернета.

**HTTPS и TLS Handshake:**
Это тот же HTTP, но обернутый в зашифрованный туннель (TLS). При установке соединения сервер и клиент договариваются о шифровании (Handshake).
*Пример:* Сервер показывает свой паспорт (SSL-сертификат), клиент проверяет его подлинность. Затем они используют *асимметричное* шифрование (публичный/приватный ключ), чтобы безопасно передать друг другу общий секретный ключ. А далее общаются уже с помощью быстрого *симметричного* шифрования.

---

**Итог:**

* **Уровни:** Сеть делится на абстракции. L4 (TCP/UDP) отвечает за логистику байтов, L7 (HTTP) дает им смысл.
* **TCP vs UDP:** TCP — надежность и порядок (для данных и транзакций). UDP — скорость любой ценой, допустимость потерь (для реалтайма и стриминга).
* **DNS:** Критичный узел инфраструктуры, переводящий имена в IP; первая точка возможной балансировки (GeoDNS).
* **HTTP:** Развивается в сторону устранения блокировок. HTTP/1.1 (текст, очередь) -> HTTP/2 (бинарный, мультиплексирование по TCP) -> HTTP/3 (мультиплексирование по быстрому UDP/QUIC).
