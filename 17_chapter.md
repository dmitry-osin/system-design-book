**Глава 17. Устойчивость (Resiliency)**

Главная задача инженера здесь — предотвратить **Каскадный сбой** (Cascading Failure), когда падение одного неважного сервиса (например, генератора аватарок) тянет за собой на дно сервис авторизации, а за ним и весь сайт.

#### 1. Circuit Breaker (Предохранитель)

*Аналогия:* Электрический щиток в квартире. Если происходит короткое замыкание в розетке, автомат "выбивает" (разрывает цепь), чтобы не сгорела вся квартира.

Если наш сервис Заказов обращается к внешнему сервису Платежей, а тот "лежит" и отвечает таймаутами по 30 секунд, наши потоки (threads) будут висеть в ожидании. Через пару минут у нас закончатся потоки, и сервис Заказов тоже умрет.

**Как работает Circuit Breaker (Паттерн имеет 3 состояния):**

1. **Closed (Закрыт - всё хорошо):** Запросы идут как обычно. Предохранитель считает ошибки.
2. **Open (Открыт - авария!):** Если процент ошибок (или таймаутов) превысил порог (например, 50% за последние 10 секунд), цепь разрывается. Теперь сервис Заказов *даже не пытается* стучаться в Платежи. Он мгновенно (Fast Fail) возвращает пользователю ошибку "Оплата временно недоступна", спасая свои потоки и не добивая лежащий сервис Платежей.
3. **Half-Open (Полуоткрыт - проверка):** Через минуту предохранитель пропускает *один* тестовый запрос. Если он успешен — цепь закрывается (возвращаемся к норме). Если опять ошибка — цепь снова открывается на минуту.

**Пример на Java (Библиотека Resilience4j):**

```java
import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;
import org.springframework.stereotype.Service;

@Service
public class PaymentClient {

    // Если этот метод начнет падать, Spring автоматически "выбьет пробки"
    // и начнет вызывать метод paymentFallback() вместо реального сетевого запроса.
    @CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
    public String processPayment(Order order) {
        // ... тяжелый HTTP вызов к сервису платежей ...
        return restTemplate.postForObject("http://payment-service/api/pay", order, String.class);
    }

    // Запасной план (Fast Fail или деградация функционала)
    public String paymentFallback(Order order, Exception ex) {
        return "Платежный шлюз недоступен. Заказ сохранен, мы спишем деньги позже.";
    }
}

```

#### 2. Bulkhead (Переборки)

*Аналогия:* Титаник утонул, потому что вода переливалась из одного отсека в другой. У современных кораблей отсеки загерметизированы (Bulkheads). Если пробьют один, корабль поплывет дальше.

В Java каждый микросервис имеет пул потоков (например, 200 потоков в Tomcat). Если сервис Заказов ходит в 3 других сервиса: `Склад`, `Платежи` и `Уведомления`. Если сервис `Уведомлений` начнет тормозить, все 200 потоков забьются ожиданием ответа от `Уведомлений`. И мы не сможем обрабатывать `Склад` и `Платежи`!

*Решение:* Изолировать пулы потоков. Выделяем 150 потоков на Платежи, 40 на Склад и жестко лимитируем всего 10 потоков на некритичные Уведомления. Если Уведомления упадут, зависнут только эти 10 потоков, а остальные 190 продолжат обрабатывать деньги!

#### 3. Retry, Exponential Backoff и Jitter

Если произошла кратковременная сетевая ошибка, логично повторить запрос (Retry). Но делать это нужно умно.

* **Глупый Retry:** Сервис упал, и 10 000 клиентов начинают долбить его запросами каждую секунду. Когда сервис попытается подняться, эта толпа (Thundering Herd) мгновенно убьет его снова (DDoS атака своими же руками).
* **Exponential Backoff (Экспоненциальная задержка):** Мы увеличиваем паузу между попытками. 1-я попытка через 1 сек, 2-я через 2 сек, 3-я через 4 сек, 4-я через 8 сек. Мы даем упавшему сервису "подышать" и восстановиться.
* **Jitter (Джиттер / Рассинхронизация):** Даже если мы ждем 8 секунд, все 10 000 клиентов проснутся ровно через 8 секунд и ударят одновременно. Jitter добавляет случайную погрешность. Клиент А ждет 8.1 сек, Клиент Б — 7.5 сек, Клиент В — 8.8 сек. Нагрузка размазывается.

#### 4. Rate Limiting (Ограничение запросов)

Как защитить *свой* API от парсеров, злых ботов или просто слишком активных клиентов? Нужно резать трафик. Есть 3 классических алгоритма:

1. **Token Bucket (Корзина с токенами):** У пользователя есть "ведро" емкостью 10 токенов. Каждый запрос забирает 1 токен. Раз в минуту система докладывает в ведро 5 токенов. Если ведро пустое — возвращаем `HTTP 429 Too Many Requests`. Позволяет обрабатывать короткие "всплески" (bursts) трафика.
2. **Leaky Bucket (Дырявое ведро):** Запросы падают в очередь (ведро). Из дырки на дне ведра запросы уходят на обработку со **строго постоянной скоростью** (например, 2 запроса в секунду). Сглаживает нагрузку на базу данных. Если ведро переполнилось — новые запросы отбрасываются.
3. **Sliding Window (Скользящее окно):** Логирует `timestamp` каждого запроса пользователя в Redis. Считает, сколько запросов было ровно за последние 60 секунд от текущей миллисекунды. Самый точный, но жрет много памяти.

**Итог Главы 17:**
Для выживания в хаосе микросервисов мы изолируем сбои с помощью **Bulkheads**, отключаем неработающие зависимости через **Circuit Breakers**, не добиваем восстанавливающиеся сервисы благодаря **Exponential Backoff с Jitter**, и защищаем себя от внешних перегрузок с помощью **Rate Limiting**.
