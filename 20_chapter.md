### Уровень 5: Экспертный уровень и специализации

*Темы для проектирования специфических и сверхнагруженных систем.*

**Глава 20. Big Data и потоковая обработка**

Есть два фундаментальных подхода к обработке данных: Пакетный (Batch) и Потоковый (Stream).

* *Аналогия:* **Batch** — это стирать всю грязную одежду разом в конце недели. **Stream** — это стирать каждую футболку вручную ровно в ту секунду, когда ты поставил на нее пятно.

#### 1. Пакетная обработка (Batch Processing)

Пакетная обработка работает с историческими, "мертвыми" данными. Данные уже лежат на диске (например, в Amazon S3 или HDFS), их размер известен, и они никуда не убегают.

**MapReduce и Hadoop:**
Исторически первым стандартом Big Data был Hadoop с его алгоритмом MapReduce. Как отсортировать или подсчитать слова в файле размером 10 Терабайт? На один сервер он не влезет.

Алгоритм делит задачу на две фазы:

1. **Map (Отображение):** Главный сервер (JobTracker) режет огромный файл на блоки по 128 МБ и раздает их тысяче дешевых серверов. Каждый сервер берет свой кусок и выполняет простую функцию фильтрации или трансформации.
2. **Reduce (Свертка):** Результаты работы тысяч серверов группируются по ключу и отправляются на несколько серверов-редукторов, которые складывают (агрегируют) финальный результат.

* **Плюсы:** Невероятная пропускная способность (Throughput). Можно переварить эксабайты данных.
* **Минусы:** Огромная задержка (Latency). Отчет генерируется часами.

#### 2. Потоковая обработка (Stream Processing)

Потоковая обработка работает с "живыми", бесконечными данными, которые летят прямо сейчас (например, через Apache Kafka). Мы не можем сказать "подожди, пока данные закончатся, чтобы посчитать сумму", потому что поток не закончится никогда.

**Окна (Windowing):**
Чтобы считать агрегации (суммы, средние) в бесконечном потоке, мы режем время на "окна":

* **Tumbling Window (Кувыркающееся окно):** Строгие отрезки без пересечений. Например, с 12:00 до 12:05, затем с 12:05 до 12:10.
* **Sliding Window (Скользящее окно):** Окно размером 5 минут, которое сдвигается каждую 1 минуту. (Полезно для поиска трендов: "количество ошибок за последние 5 минут").

**Инструменты:**

* **Apache Spark (Streaming):** На самом деле это "Микро-батчинг". Spark накапливает события за 1 секунду, а потом обрабатывает эту секунду как малюсенький Batch. Это быстро, но задержка всегда будет около 1-2 секунд.
* **Apache Flink:** Истинный потоковый движок. Он обрабатывает каждое событие поштучно (Event-at-a-time). Задержка (Latency) измеряется миллисекундами. Идеален для антифрода в финтехе.
* **Kafka Streams:** Легковесная библиотека прямо для твоего Java-приложения. Не нужен отдельный кластер (как для Flink или Spark).

**Пример на Java (Kafka Streams — подсчет слов в реальном времени):**

```java
import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.kstream.KStream;
import org.apache.kafka.streams.kstream.KTable;
import java.util.Arrays;

public class WordCountStream {

    public void buildTopology(StreamsBuilder builder) {
        // 1. Читаем бесконечный поток предложений из топика "text-input"
        KStream<String, String> textLines = builder.stream("text-input");

        // 2. Магия потоковой трансформации
        KTable<String, Long> wordCounts = textLines
            // Разбиваем предложения на отдельные слова
            .flatMapValues(textLine -> Arrays.asList(textLine.toLowerCase().split("\\W+")))
            // Группируем по самому слову
            .groupBy((key, word) -> word)
            // Считаем количество (под капотом Kafka Streams сохранит стейт на локальный диск)
            .count();

        // 3. Результат непрерывно пишется в другой топик "word-count-output"
        wordCounts.toStream().to("word-count-output");
    }
}

```

#### 3. Архитектуры: Lambda vs Kappa

Как объединить точность исторических данных со скоростью реального времени? Для этого придумали две глобальные архитектуры.

**Lambda Architecture (Лямбда-архитектура):**
Состоит из трех слоев:

1. **Batch Layer (Холодный слой):** Hadoop/Spark складывает абсолютно все сырые данные в огромное озеро (Data Lake). Раз в сутки он пересчитывает 100% точные агрегации. Это долго, но математически непогрешимо.
2. **Speed Layer (Горячий слой):** Flink/Storm слушает Kafka и на лету считает приблизительные результаты за сегодняшний день.
3. **Serving Layer (Слой выдачи):** Когда пользователь запрашивает аналитику, сервис делает запрос в оба слоя. Вчерашние данные он берет из точного Batch Layer, а данные за последние 2 часа — из быстрого Speed Layer. Затем он сливает их вместе.

* *Проблема:* Тебе нужно писать и поддерживать **два разных кода** (один для Batch на Spark, другой для Stream на Flink), которые делают логически одно и то же.

**Kappa Architecture (Каппа-архитектура):**
Эволюция Лямбды, которая говорит: **"Batch — это просто поток, который имеет начало и конец. Давайте всё сделаем потоком!"**

1. Мы выкидываем Batch Layer и Hadoop.
2. Все данные (даже за 10 лет) мы храним в бесконечной Apache Kafka.
3. Мы используем один потоковый движок (Apache Flink).
4. Если нам нужно пересчитать старые данные, мы просто заставляем Flink прочитать Kafka с нулевого оффсета (с самого начала времен) на максимальной скорости.

* *Плюсы:* Один стек технологий (Kafka + Flink), одна кодовая база.
* *Где применяется:* Uber, LinkedIn, современные Cloud-Native дата-платформы.

---

**Итог:**

* **Batch Processing** (MapReduce, Hadoop) перемалывает колоссальные объемы сохраненных данных с высокой пропускной способностью, но огромной задержкой.
* **Stream Processing** (Flink, Kafka Streams) обрабатывает бесконечные потоки данных в ту же миллисекунду, когда они появились, используя концепцию Окон (Windows).
* **Lambda Architecture** спаривает медленный, но точный Batch с быстрым Stream, заставляя поддерживать два разных стека.
* **Kappa Architecture** упрощает систему, превращая всё в стриминг, где даже исторические данные проигрываются через один и тот же потоковый движок (Flink) из лога Kafka.
