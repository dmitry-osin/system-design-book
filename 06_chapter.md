### Уровень 2: Компоненты системы (Core Building Blocks)

*Инструментарий архитектора для решения типовых задач.*

**Глава 6. Кэширование (Caching)**

Кэширование — это сохранение результатов дорогостоящих вычислений или частых запросов к базе данных в сверхбыструю память (обычно в оперативную память — RAM).
*Аналогия:* Если тебя спросят, сколько будет 2 + 2, ты ответишь сразу (из кэша в голове). Если спросят 156 * 48, ты возьмешь калькулятор (пойдешь в базу данных). В следующий раз, если тебя снова спросят 156 * 48, ты уже вспомнишь ответ, не считая заново.

#### 1. Стратегии кэширования

Как именно приложение должно общаться с кэшем и базой данных? Есть 4 классических паттерна.

* **Cache-Aside (Lazy Loading):** Самый популярный паттерн. Приложение сначала идет в кэш. Если данных там нет (Cache Miss), оно идет в базу, забирает данные, **само** кладет их в кэш и возвращает пользователю.
* *Плюсы:* В кэше лежит только то, что реально запрашивают.
* *Минусы:* Первый запрос всегда медленный (ведь нужно сходить и в пустой кэш, и в базу).


* **Write-Through (Сквозная запись):** Когда пользователь обновляет профиль, приложение пишет новые данные синхронно и в кэш, и в базу. Запрос считается успешным только когда обновлены обе системы.
* *Плюсы:* Данные в кэше никогда не устаревают.
* *Минусы:* Медленная запись, так как нужно дождаться ответа от двух систем.


* **Write-Back / Write-Behind (Отложенная запись):** Приложение пишет данные **только в кэш** (например, в Redis) и сразу отвечает пользователю "Успешно". Кэш сам, асинхронно, раз в пару секунд сбрасывает эти данные в тяжелую БД.
* *Плюсы:* Невероятно быстрая запись (идеально для лайков, счетчиков просмотров YouTube).
* *Минусы:* Огромный риск. Если сервер кэша перезагрузится до того, как сбросит данные в БД — данные потеряны навсегда.


* **Write-Around:** Приложение пишет данные сразу в БД, в обход кэша. А читает через Cache-Aside.
* *Где применять:* Для данных, которые пишутся один раз и почти никогда не читаются сразу (например, логи аудита или загрузка архива). Нет смысла забивать дорогую оперативную память кэша тем, что не будут читать.



**Пример на Java (Паттерн Cache-Aside):**

```java
import org.springframework.stereotype.Service;

@Service
public class UserService {

    private final UserRepository db;    // Наш PostgreSQL
    private final RedisClient redis;    // Наш Кэш

    public UserProfile getUserProfile(String userId) {
        String cacheKey = "user:profile:" + userId;

        // 1. Пытаемся взять данные из быстрого кэша (Redis)
        UserProfile cachedProfile = redis.get(cacheKey);
        if (cachedProfile != null) {
            return cachedProfile; // Cache Hit! Возвращаем за 2 миллисекунды.
        }

        // 2. Cache Miss. Идем в медленную базу данных.
        UserProfile profileFromDb = db.findById(userId)
            .orElseThrow(() -> new UserNotFoundException());

        // 3. Кладем результат в кэш, чтобы следующий запрос был быстрым.
        // Обязательно ставим TTL (Time To Live), например 1 час (3600 секунд),
        // чтобы кэш не разросся до бесконечности.
        redis.setEx(cacheKey, 3600, profileFromDb);

        return profileFromDb;
    }
}

```

#### 2. Уровни кэширования

Чтобы запрос даже не доходил до твоего бэкенда, кэшировать можно на разных этапах пути пользователя:

1. **Client-side (Браузер/Приложение):** Через HTTP-заголовки (например, `Cache-Control: max-age=3600`) мы говорим браузеру: "Эта картинка не изменится час, не скачивай ее заново". Самый дешевый вид кэша — он вообще не нагружает наши серверы.
2. **CDN (Content Delivery Network):** Географически распределенные серверы (Edge-серверы). Если сервер в Москве, а пользователь в Сиднее, картинка будет грузиться долго. CDN сохраняет статику (картинки, CSS, JS) на сервере в Сиднее и отдает ее локально.
3. **Load Balancer (Reverse Proxy):** Nginx умеет кэшировать целые страницы или ответы API. Если 1000 человек запрашивают `/api/news/top`, Nginx может сходить на твой бэкенд один раз, а остальным 999 отдать закэшированный JSON прямо с балансировщика.
4. **Server-side:**
* *Local Cache:* Кэш внутри памяти самого приложения (библиотеки Caffeine, Guava в Java). Очень быстрый, но если у тебя 10 инстансов приложения, у каждого будет свой локальный кэш (возможна рассинхронизация).
* *Distributed Cache:* Внешний сервер (Redis, Memcached), к которому обращаются все твои инстансы.


5. **Database Cache:** Сама база данных (например, Postgres) держит в оперативной памяти часто используемые страницы данных (`shared_buffers`), чтобы не читать их с диска.

#### 3. Проблемы кэширования (И почему это больно)

В Computer Science есть известная шутка: *"Есть только две сложные проблемы: инвалидация кэша и придумывание названий переменным"*.

* **Cache Invalidation (Инвалидация кэша):** Как понять, что данные в кэше устарели?
* Самый простой путь — использовать **TTL (Time To Live)**. Данные сами удалятся через час.
* Сложный путь — когда пользователь меняет аватарку, твой код обновления должен принудительно удалить (эвиктить) старую запись из Redis. Если этот шаг сломается, пользователь будет видеть старую аватарку.


* **Eviction policies (Политики вытеснения):** Оперативная память ограничена. Когда Redis заполняется на 100%, кого удалять?
* **LRU (Least Recently Used):** Удаляем то, что дольше всего не запрашивали. Самый популярный и логичный алгоритм.
* **LFU (Least Frequently Used):** Удаляем то, что запрашивают реже всего в целом.


* **Thundering Herd problem (Эффект толпы / Cache Stampede):** Представь, что мы закэшировали профиль Криштиану Роналду (миллионы просмотров в минуту) с TTL на 1 час.
Ровно через час кэш испаряется. В эту же миллисекунду 50 000 запросов не находят профиль в кэше (Cache Miss) и **одновременно** бьют в базу данных. База падает от перегрузки.
* *Решение:* Использование распределенных блокировок (Mutex/Lock). Когда кэш протухает, только *первый* запрос получает право сходить в базу. Остальные 49 999 запросов ждут долю секунды, пока первый не положит свежие данные обратно в кэш.



---

**Итог:**
Кэширование — главный инструмент снижения нагрузки и ускорения ответа (Latency). Мы выбираем стратегию (чаще всего Cache-Aside для чтения и Write-Through для надежной записи), расставляем кэши на разных уровнях (от браузера до Redis) и обязательно защищаемся от "Эффекта толпы", чтобы протухание популярного ключа не убило нашу базу данных.
