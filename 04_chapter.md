**Глава 4. Базовое масштабирование**

Масштабирование — это процесс увеличения вычислительных мощностей системы для обработки возросшей нагрузки. Глобально у инженера есть только два пути.

#### 1. Vertical Scaling (Scale Up) vs Horizontal Scaling (Scale Out)

**Вертикальное масштабирование (Scale Up):**
Это подход «купить сервер помощнее». У тебя был сервер на 4 ядра процессора и 16 ГБ оперативной памяти. Ты останавливаешь его, платишь хостинг-провайдеру больше денег и запускаешь сервер на 64 ядра и 256 ГБ памяти.

* **Плюсы:** Не нужно менять архитектуру и переписывать код. Всё работает как раньше, просто быстрее.
* **Минусы:** Есть физический предел (нельзя купить сервер с бесконечной памятью). Это дорого (топовое железо стоит непропорционально дорого). У тебя остается **единая точка отказа (Single Point of Failure - SPOF)**: если этот супер-сервер сгорит, упадет весь бизнес.
* *Где применяется:* Чаще всего так масштабируют классические реляционные базы данных (PostgreSQL) на начальных этапах, потому что их горизонтально масштабировать (шардировать) очень больно.

**Горизонтальное масштабирование (Scale Out):**
Это подход «нанять больше рабочих». Вместо одного супер-сервера ты ставишь 10 дешевых средних серверов.

* **Плюсы:** Бесконечное масштабирование (просто добавляй сервера). Отказоустойчивость: если 2 сервера из 10 сгорят, пользователи этого даже не заметят, система продолжит работать на оставшихся 8.
* **Минусы:** Архитектура резко усложняется. Твое приложение обязано быть **Stateless (без сохранения состояния)**.

> **Важное правило System Design:** Чтобы легко масштабироваться горизонтально, сервер приложений не должен хранить сессии пользователей в своей локальной памяти. Иначе пользователь авторизуется на Сервере 1, следующий его запрос попадет на Сервер 2, и его выкинет из аккаунта. Состояние нужно выносить во внешние хранилища (например, в Redis).

#### 2. Load Balancing (Балансировка нагрузки)

Если у нас 10 одинаковых серверов приложения (мы применили Scale Out), как пользователь узнает, к какому из них подключаться? Ему нужен "регулировщик". Эту роль выполняет **Load Balancer (LB)**.

Пользователь отправляет запрос на IP-адрес балансировщика, а тот уже решает, какому конкретно серверу (из пула доступных) передать эту задачу.

**L4 vs L7 Балансировка (вспоминаем Главу 1):**

* **L4 (Транспортный уровень):** Балансировщик смотрит только на IP-адрес и порт (TCP/UDP). Он не знает, что внутри пакета (картинка это или запрос к API). Он работает невероятно быстро, просто перекидывая сетевые пакеты.
* *Инструменты:* AWS Network Load Balancer, HAProxy.


* **L7 (Прикладной уровень):** Балансировщик читает содержимое HTTP-запроса. Он видит URL, заголовки, куки. Это позволяет делать "умную" маршрутизацию. Например, если в URL есть `/api/video/*`, он отправит запрос на сервера, оптимизированные для отдачи видео, а если `/api/payments/*` — на защищенные биллинговые сервера.
* *Инструменты:* Nginx, AWS Application Load Balancer, Traefik.



#### 3. Алгоритмы балансировки нагрузки

Как именно балансировщик решает, куда отправить следующий запрос? Для этого есть классические алгоритмы (их часто просят реализовать на собеседованиях):

* **Round Robin (Карусель):** Самый простой. Запросы раздаются по кругу: первому серверу, второму, третьему, потом снова первому. Идеально, если все сервера одинаковой мощности и запросы требуют примерно одинакового времени на обработку.
* **Weighted Round Robin:** То же самое, но с "весами". Если Сервер 1 в два раза мощнее Сервера 2, мы даем ему вес 2 и он будет получать два запроса на каждый один запрос ко второму серверу.
* **Least Connections (Наименьшее число подключений):** Балансировщик считает, сколько активных соединений сейчас висит на каждом сервере. Новый запрос улетит на тот сервер, который сейчас наиболее свободен. Спасает, когда некоторые запросы обрабатываются долго и "забивают" конкретный узел.
* **IP Hash (Хэширование по IP):** Балансировщик берет IP-адрес пользователя (например, `192.168.1.50`), вычисляет из него хэш и на основе этого числа привязывает пользователя к конкретному серверу.
* *Зачем это нужно:* Гарантирует, что конкретный клиент *всегда* будет попадать на один и тот же сервер. Это полезно, если мы работаем с кэшем в памяти сервера или с WebSockets.



**Пример (Псевдокод алгоритма Round Robin на Java):**
Представь, что тебе на лайв-кодинге дали задачу написать простейший балансировщик.

```java
import java.util.List;
import java.util.concurrent.atomic.AtomicInteger;

public class RoundRobinLoadBalancer {
    // Список IP-адресов наших серверов
    private final List<String> servers;
    
    // Атомарный счетчик для потокобезопасности, 
    // ведь к балансировщику обращаются тысячи потоков одновременно
    private final AtomicInteger currentIndex;

    public RoundRobinLoadBalancer(List<String> servers) {
        this.servers = servers;
        this.currentIndex = new AtomicInteger(0);
    }

    // Метод, который вернет IP сервера для очередного запроса
    public String getNextServer() {
        if (servers == null || servers.isEmpty()) {
            throw new IllegalStateException("Нет доступных серверов!");
        }
        
        // Получаем текущее значение и сразу увеличиваем на 1
        int index = currentIndex.getAndIncrement();
        
        // Используем остаток от деления (modulo), чтобы зациклить обход
        // Например, если серверов 3, а index стал 5: 5 % 3 = 2 (третий сервер)
        int serverIndex = Math.abs(index % servers.size());
        
        return servers.get(serverIndex);
    }
}

```

#### 4. Software vs Hardware Load Balancers

* **Software (Программные):** Это обычные программы (Nginx, HAProxy, Envoy), которые ты устанавливаешь на обычный Linux-сервер. Они дешевые, гибкие, их легко настраивать как код (Infrastructure as Code). Используются в 99% современных веб-проектов.
* **Hardware (Аппаратные):** Физические железные коробки (например, от F5 Networks или Citrix), созданные специально только для одной цели — балансировать трафик. В них используются специальные чипы (ASIC), которые умеют парсить сеть на аппаратном уровне. Стоят сотни тысяч долларов, используются в корпоративных дата-центрах и телеком-компаниях для пропуска терабит трафика без задержек.

---

**Итог:**

* Вертикальное масштабирование (Scale Up) упирается в потолок железа и создает точку отказа.
* Горизонтальное масштабирование (Scale Out) — стандарт индустрии, но требует архитектуры без сохранения состояния (Stateless) на серверах.
* При горизонтальном масштабировании обязателен Load Balancer. L4 быстрее, L7 умнее (маршрутизация по URL).
* Алгоритмы балансировки (Round Robin, Least Connections, IP Hash) подбираются под характер нагрузки (ровная, тяжелая или требующая "липкости" сессии к одному серверу).
