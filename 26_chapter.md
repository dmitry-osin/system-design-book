**Глава 26. Классические кейсы для разбора**

Поскольку мы подошли к финалу, разберем абсолютную классику, на которой отлично проверяется понимание баз данных, кэшей и генерации ID. Мы применим фреймворк из 25 главы.

#### Разбор кейса 1: Проектирование URL Shortener (аналог bit.ly)

**Шаг 1. Выяснение требований**

* **Функциональные:**
1. Пользователь передает длинный URL и получает короткий (например, `[tiny.com/aBc12x](https://tiny.com/aBc12x)`).
2. При переходе по короткому URL происходит редирект на оригинальный.
3. Ссылки живут вечно.


* **Нефункциональные:**
1. Высокая доступность (High Availability) — если сервис редиректа упадет, сломаются миллионы ссылок в интернете.
2. Чтений в 100 раз больше, чем записей (Read-heavy).
3. Задержка редиректа должна быть минимальной (< 50 мс).



**Шаг 2. Оценка нагрузок (Back-of-the-envelope)**

* *Запись:* Допустим, мы генерируем 100 миллионов новых ссылок в месяц.
$100,000,000 / (30 \text{ дней} \times 24 \text{ часа} \times 3600 \text{ секунд}) \approx 40$ записей в секунду (Write QPS).
* *Чтение:* Соотношение 100:1. Значит, $40 \times 100 = 4000$ чтений в секунду (Read QPS). Пиковая нагрузка может быть в 5 раз выше = $20,000$ QPS.
* *Хранилище:* Мы храним ссылки 10 лет. $100 \text{ млн} \times 12 \text{ мес} \times 10 \text{ лет} = 12 \text{ миллиардов}$ записей.
Если одна запись (Длинный URL + Короткий хэш + Дата) весит 500 байт, то нам нужно $12 \text{ млрд} \times 500 \text{ байт} = 6 \text{ Терабайт}$ дискового пространства.

**Шаг 3. High-level дизайн**

1. **Пользователь** отправляет длинный URL на **API Gateway (Load Balancer)**.
2. Балансировщик направляет запрос на **Web Server**.
3. Сервер общается с генератором уникальных ID, создает короткую ссылку и сохраняет пару `(Короткая, Длинная)` в **Базу Данных**.
4. При чтении Web Server сначала проверяет **Кэш (Redis)**. Если там пусто — идет в БД.

**Шаг 4. Детальный дизайн (Магия под капотом)**

**А. Длина хэша и алгоритм кодирования**
Из каких символов состоит короткая ссылка? Обычно это `[a-z, A-Z, 0-9]`. Итого $26 + 26 + 10 = 62$ возможных символа (Base62 кодирование).
Какой длины должен быть хэш, чтобы вместить наши 12 миллиардов ссылок?

* $62^6 = 56 \text{ миллиардов}$ комбинаций.
* $62^7 = 3.5 \text{ триллиона}$ комбинаций.
Длины в 7 символов нам хватит за глаза на сотни лет работы.

**Б. Как сгенерировать 7 символов?**

* *Плохой подход (Хэш-функция MD5):* Взять длинный URL, прогнать через MD5 (получим 32 символа) и отрезать первые 7. Проблема: Коллизии! Два разных URL могут дать одинаковые 7 первых символов. Придется ходить в БД, проверять наличие коллизии, добавлять соль, снова хэшировать... Это убьет производительность.
* *Хороший подход (Twitter Snowflake + Base62):*
1. Берем наш проверенный генератор ID из Главы 9 (Snowflake). Он за 1 микросекунду выдает нам уникальное 64-битное число, например `2009215674938`.
2. Мы переводим это число из десятичной системы счисления в систему счисления по основанию 62 (Base62).
3. Получаем строку `cX3yA`. Это и есть наша уникальная ссылка! Никаких коллизий быть не может математически, так как Snowflake выдает только уникальные числа.



*Пример на Java (Кодирование числа в Base62):*

```java
public class Base62Encoder {
    private static final String ALPHABET = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    private static final int BASE = ALPHABET.length(); // 62

    // Превращаем уникальный ID из Snowflake в короткую строку
    public static String encode(long num) {
        if (num == 0) return String.valueOf(ALPHABET.charAt(0));
        StringBuilder sb = new StringBuilder();
        while (num > 0) {
            int remainder = (int) (num % BASE);
            sb.append(ALPHABET.charAt(remainder));
            num /= BASE;
        }
        return sb.reverse().toString();
    }
}

```

**В. База Данных**
Нам нужно хранить 6 Терабайт простых данных без сложных связей (JOIN-ов).
Отличным выбором будет NoSQL Key-Value хранилище (например, **Amazon DynamoDB** или **Cassandra**).

* *Primary Key (Partition Key):* Короткий URL. Это обеспечит нам идеальное хэш-шардирование (Consistent Hashing из Главы 14). Данные равномерно размажутся по всем серверам БД, и поиск длинного URL по короткому будет занимать $O(1)$.

**Г. Кэширование (Ускоряем редирект)**
$20,000$ чтений в секунду — это терпимо для Cassandra, но мы хотим задержку < 10 мс. Ставим **Redis**.

* *Стратегия вытеснения:* **LRU (Least Recently Used)**. Вспоминаем правило Парето: 80% трафика приходится на 20% ссылок (вирусные твиты, свежие новости). Нам не нужно держать все 6 ТБ в оперативной памяти. Хватит кластера Redis на 100-200 ГБ, чтобы кэшировать только самые "горячие" ссылки.

**Д. HTTP Редиректы (Скрытая ловушка собеседования)**
Когда сервер находит длинный URL, с каким HTTP-кодом он должен ответить браузеру?

* **301 (Moved Permanently):** Браузер закэширует этот ответ. В следующий раз, когда пользователь кликнет на эту же короткую ссылку, браузер **вообще не пойдет на наш сервер**, а сразу перейдет на длинный URL. Это колоссально снижает нагрузку на нас. Но мы теряем возможность собирать аналитику (сколько раз кликнули).
* **302 (Found / Temporary Redirect):** Браузер не кэширует редирект. Каждый клик всегда проходит через наш сервер. Нагрузка выше, зато мы можем собирать точную статистику кликов для бизнес-аналитики.
*Вывод:* В реальных коммерческих системах выбирают 302 ради аналитики.
