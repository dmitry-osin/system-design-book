**Глава 7. Асинхронность и Очереди сообщений**

Суть асинхронности в System Design: "Я отдал команду, и мне не нужно ждать результата прямо сейчас. Я уверен, что система когда-нибудь ее выполнит".

#### 1. Message Queues (Брокеры сообщений): RabbitMQ

Брокер сообщений — это умный почтальон. Сервер заказов (Producer) просто кидает в RabbitMQ сообщение: "Заказ #123 создан". И моментально отвечает пользователю: "Заказ оформлен!". Дальше RabbitMQ сам раздает это сообщение нужным сервисам (Consumers).

У RabbitMQ есть две главные модели работы:

* **Point-to-Point (Рабочие очереди):** У нас есть одна очередь сообщений и 5 одинаковых воркеров (например, сервисов генерации PDF). Когда приходит задача, RabbitMQ отдает её **только одному** свободному воркеру. Как только воркер сделал работу, он шлет подтверждение (ACK), и сообщение удаляется из брокера навсегда.
* *Аналогия:* Очередь на кассу в супермаркете. Один покупатель — один кассир.


* **Publish/Subscribe (Pub/Sub):** Сообщение отдается всем заинтересованным. Сервер заказов кидает событие в "обменник" (Exchange), а тот копирует его во множество очередей. Сервер склада забирает свою копию, почтовый сервис — свою копию.
* *Аналогия:* Радиовещание. Диктор говорит один раз, а слышат все, кто настроился на волну.



**Главная особенность RabbitMQ:** Умный брокер, глупые клиенты. RabbitMQ сам следит за тем, кто прочитал сообщение, и **удаляет** его после прочтения.

#### 2. Event Streaming: Apache Kafka (Это не очередь!)

Часто на собеседованиях спрашивают: "В чем разница между RabbitMQ и Kafka?". Это фундаментально разные вещи.

Kafka — это не почтальон. Kafka — это **неизменяемый журнал (Log)**.
Когда сервис пишет сообщение в Kafka, оно прикрепляется в конец файла на жестком диске. И оно **не удаляется** после того, как его прочитали (оно хранится заданное время, например, 7 дней, или пока не кончится место).

* **Топики и Партиции:** Сообщения лежат в "Топиках" (Topics). Чтобы Kafka могла переваривать миллионы сообщений в секунду, топик пилится на куски — Партиции (Partitions), которые лежат на разных серверах кластера.
* **Оффсеты (Offsets):** В Kafka брокер (сервер) "глупый". Он не запоминает, кто что прочитал. Клиент (Consumer) должен быть "умным". Он сам запоминает свой номер строки (Offset), на котором остановился.
* *Зачем это нужно:* Если у тебя упал сервис аналитики, он может подняться, перемотать свой оффсет на 2 часа назад и прочитать все события заново! В RabbitMQ так сделать нельзя — сообщения уже удалены.



#### 3. Критические концепции обработки: Идемпотентность и DLQ

При работе с асинхронностью мы теряем 100% гарантию надежности сети. Возникает проблема: **"At-least-once delivery"** (Доставка как минимум один раз).

Представь ситуацию:

1. Почтовый сервис прочитал из RabbitMQ задачу "Отправить письмо".
2. Отправил письмо пользователю.
3. Попытался отправить RabbitMQ сигнал "Успех (ACK)", но в этот момент моргнула сеть (Network timeout).
4. RabbitMQ думает, что сервис не справился, и отдает эту задачу другому воркеру.
5. Пользователь получает **два** письма (или у него списывают деньги дважды!).

**Идемпотентность:**
Это свойство системы, при котором многократное применение одной и той же операции дает тот же результат, что и однократное. Мы **обязаны** делать консьюмеры идемпотентными.
Обычно это решается через `Idempotency Key` (Уникальный ID сообщения), который мы сохраняем в базу данных.

**Dead Letter Queues (DLQ - Очередь недоставленных писем):**
Что делать, если мы получили сообщение с кривым JSON и наш парсер падает с ошибкой? Если мы просто вернем ошибку в RabbitMQ, он будет пытаться отправить это сообщение снова и снова бесконечно (Poison Pill).
*Решение:* Если мы пытались обработать сообщение 3 раза и не смогли, мы перекидываем его в специальную очередь DLQ. Там оно лежит, пока программист не придет и руками не разберется, что пошло не так.

**Пример на Java (Идемпотентный Consumer):**

```java
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class PaymentConsumer {

    private final ProcessedMessageRepository messageRepo;
    private final PaymentService paymentService;

    // Метод, который слушает очередь RabbitMQ / Kafka
    @RabbitListener(queues = "payment_commands")
    @Transactional // Обязательно в одной транзакции с бизнес-логикой
    public void receivePaymentCommand(PaymentCommand command) {
        
        // 1. Проверяем Идемпотентность. 
        // messageId (UUID) генерируется на стороне отправителя ОДИН раз.
        String messageId = command.getMessageId();
        
        if (messageRepo.existsById(messageId)) {
            // Мы уже обрабатывали это сообщение! (Сеть моргнула при ACK)
            // Просто игнорируем его и говорим брокеру "Все ок", чтобы он его удалил.
            System.out.println("Дубликат сообщения обнаружен и проигнорирован: " + messageId);
            return;
        }

        // 2. Выполняем бизнес-логику (списываем деньги)
        paymentService.chargeUser(command.getUserId(), command.getAmount());

        // 3. Запоминаем, что мы успешно обработали этот messageId.
        // Так как мы в @Transactional, если chargeUser упадет, 
        // эта запись не сохранится в БД.
        messageRepo.save(new ProcessedMessage(messageId));
        
        // По завершении метода Spring сам отправит ACK в брокер.
    }
}

```

---

**Итог:**

* Очереди — главный способ распилить монолитный синхронный процесс на быстрые независимые куски.
* **RabbitMQ** (Message Queue) — маршрутизирует и удаляет сообщения. Отлично подходит для задач ("Отправь письмо", "Сгенерируй отчет").
* **Apache Kafka** (Event Streaming) — сохраняет историю событий в виде лога на диске. Идеальна для аналитики, трансляции изменений (CDC) и микросервисов, которым нужно "проигрывать" историю.
* Главное правило распределенных систем: сеть ненадежна, сообщения будут дублироваться. **Идемпотентность** (защита от двойной обработки) — обязательное требование к любому Consumer-у.
