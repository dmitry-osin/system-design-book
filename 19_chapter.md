#### 1. Стратегии безопасного деплоя (Deployment)

Деплой — это процесс доставки нового кода на сервер. Чтобы сделать это незаметно для пользователя (Zero-Downtime), мы используем умные стратегии маршрутизации трафика (на уровне балансировщика или Kubernetes).

* **Rolling Updates (Постепенное обновление):** Это стандарт в Kubernetes. Если у тебя работает 10 Подов версии `v1`, Кубернетес запускает один Под версии `v2`. Как только он сигнализирует, что готов (Healthcheck = OK), Кубернетес убивает один Под `v1`. И так по цепочке.
* *Проблема:* В течение нескольких минут в системе одновременно работают и старый, и новый код. Они обязаны быть совместимы друг с другом и с текущей схемой базы данных!


* **Blue/Green Deployment (Сине-зеленое развертывание):** Ты поднимаешь *точную копию* всего продакшена (Зеленая среда) с новой версией кода. Трафик пользователей всё еще идет на старую (Синюю) среду. Тестировщики проверяют Зеленую. Если всё ок — балансировщик по щелчку пальцев (за 1 миллисекунду) переключает 100% трафика на Зеленую среду.
* *Плюсы:* Идеальный и мгновенный откат (Rollback). Если Зеленая упала, просто переключаем балансировщик обратно на Синюю.
* *Минусы:* Тебе нужно иметь в 2 раза больше серверов. Это дорого.


* **Canary Releases (Канареечные релизы):** Назван в честь шахтеров, которые брали канареек в шахту (птичка падала в обморок при малейшей утечке газа, спасая людей). Ты пускаешь на новую версию `v2` только **1% пользователей** (или только сотрудников компании). Сидишь и смотришь в Grafana. Если метрики ошибок не поползли вверх, увеличиваешь трафик до 10%, затем до 100%.

**Feature Toggles (Фича-флаги): Разделяй Deploy и Release**
Самый мощный инструмент Senior инженера. Мы выкатываем новый код на продакшен, но он **выключен** с помощью простого `if`.

```java
@Service
public class CheckoutService {

    private final FeatureToggleService featureToggle;
    private final OldCartProcessor oldProcessor;
    private final NewFastCartProcessor newProcessor;

    public CheckoutResult processCart(Cart cart, User user) {
        // Мы можем включить новую фичу только для бета-тестеров
        // или для 5% случайных пользователей, просто поменяв конфиг в админке
        if (featureToggle.isEnabled("USE_NEW_CART_ENGINE", user.getId())) {
            return newProcessor.process(cart);
        } else {
            return oldProcessor.process(cart); // Старый надежный код
        }
    }
}

```

*Зачем это нужно:* Если новая корзина всё-таки ломается, тебе не нужно запускать тяжелый процесс отката серверов (Rollback). Ты просто нажимаешь кнопку в админке (например, в системе LaunchDarkly), переменная меняется на `false`, и баг исчезает у всех пользователей за долю секунды.

#### 2. Эволюция схемы данных (Zero-Downtime DB Migrations)

Представь популярный вопрос на собеседовании: *"У нас таблица Users на 100 миллионов строк. Есть колонка `name` (Иван Иванов). Мы хотим разбить её на две колонки `first_name` и `last_name`. Как сделать это без остановки продакшена?"*

Если ты сделаешь `ALTER TABLE ...`, база заблокирует таблицу на несколько минут (или часов). Прод ляжет.
Решение — паттерн **Expand and Contract (Расширение и Сжатие)**. Он состоит из 5 шагов, каждый из которых требует отдельного деплоя кода!

1. **Expand (БД):** Мы добавляем колонки `first_name` и `last_name`, разрешая им быть пустыми (`NULL`). Прод работает как обычно.
2. **Code (Запись в обе):** Мы деплоим код, который при создании/обновлении пользователя пишет данные и в старую колонку `name`, и в новые. Читаем мы всё еще из старой `name`.
3. **Migration (Фоновая миграция):** Мы запускаем скрипт, который по 1000 строк берет старые данные из `name`, парсит их и заполняет пустые ячейки в новых колонках. Пользователи этого не замечают.
4. **Code (Переключение чтения):** Мы деплоим код, который теперь *читает и пишет* только в новые колонки `first_name` и `last_name`. Старая колонка больше не используется кодом.
5. **Contract (Очистка):** Через неделю, когда мы убедились, что всё работает идеально, мы спокойно удаляем колонку `name` из базы.

#### 3. Версионирование API и Политики вывода из эксплуатации

В распределенных системах ты не контролируешь клиентов. Мобильное приложение версии 1.0 может жить на телефоне дедушки 3 года, и он его не обновит. Если ты сломаешь старое API, приложение дедушки перестанет работать.

* Всегда версионируй API с первого дня: `GET /api/v1/users`.
* Если меняется контракт (например, вместо списка возвращается объект), создавай `v2`. Обе версии должны работать параллельно!
* **Deprecation Policy:** Ты должен уметь отключать старье. Объявляешь, что `v1` будет отключена через 6 месяцев. Добавляешь в ответ заголовок `Warning: 299 - API Deprecated`. Затем начинаешь искусственно замедлять `v1` (Brownout) по ночам, чтобы разработчики старых клиентов забили тревогу и перешли на `v2`.

#### 4. Contract Testing (Контрактное тестирование)

В микросервисах классическое E2E (End-to-End) тестирование — это боль. Чтобы протестировать Корзину, тебе нужно поднять локально 15 баз данных и 20 смежных сервисов. Это медленно и хрупко.

Решение: **Consumer-Driven Contracts (CDC)** (например, фреймворк Pact).

* **Идея:** Сервис-потребитель (Фронтенд) пишет JSON-файл (Контракт): *"Когда я делаю GET /user/1, я ожидаю получить поле 'age' типа Integer"*.
* Этот файл кладется в специальный брокер контрактов.
* Когда Бэкенд (Провайдер) пытается слить свой код в `master`, CI/CD пайплайн скачивает этот контракт и прогоняет тесты.
* Если Бэкенд-разработчик случайно переименовал 'age' в 'userAge', CI/CD упадет с ошибкой красным цветом **до того**, как код попадет на прод. Никто не поднимает реальные базы, тесты идут миллисекунды, а контракты гарантируют, что сервисы не сломают друг друга в бою.

---

**Итог Главы 19:**

* Деплой должен быть невидимым для клиента: мы используем **Rolling** или **Blue/Green** обновления.
* Мы разделяем понятия "выкатить код" и "включить фичу" с помощью **Feature Toggles**. Это спасает при инцидентах.
* Базы данных под нагрузкой меняются не одним ALTER-запросом, а многошаговым паттерном **Expand and Contract**.
* Мы версионируем API и защищаем контракты между сервисами с помощью автоматических проверок (CDC), избавляясь от тяжелого ручного E2E тестирования.
