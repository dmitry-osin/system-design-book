**Глава 8. Хранение статики и доставка контента**

#### 1. Proxy (Reverse Proxy vs Forward Proxy)

Слово "прокси" (Proxy) означает "посредник". В архитектуре есть два совершенно разных типа посредников.

* **Forward Proxy (Прямой прокси):** Защищает и скрывает **клиента**.
* *Пример:* Корпоративный прокси в офисе или VPN. Твой компьютер делает запрос к прокси, а прокси идет в интернет. Сервер (например, Google) думает, что запрос пришел от прокси-сервера, и не знает твоего реального IP.


* **Reverse Proxy (Обратный прокси):** Защищает и скрывает **сервер**.
* *Пример:* Nginx, HAProxy. Пользователь думает, что обращается к `mysite.com`, но на самом деле он попадает на Reverse Proxy. А тот уже тайно перенаправляет запрос на один из твоих скрытых микросервисов (например, `10.0.0.5:8080`).
* *Зачем он нужен:*
1. **Единая точка входа:** Скрывает внутреннюю топологию сети.
2. **SSL Termination:** Прокси берет на себя тяжелую математику по расшифровке HTTPS. Внутренние микросервисы общаются уже по простому и быстрому HTTP.
3. **Раздача статики:** Nginx написан на C и оптимизирован отдавать статические файлы (HTML, CSS, картинки) с диска в 100 раз быстрее, чем это сделает приложение на Java или Python.





#### 2. CDN (Content Delivery Network)

Законы физики неумолимы: свет летит по оптоволокну с конечной скоростью. Если твой сервер (Origin) стоит в Москве, а пользователь открывает сайт в Австралии, каждый пакет данных будет идти туда-обратно около 300 миллисекунд. Для загрузки тяжелой страницы это выльется в секунды ожидания.

**CDN** — это глобальная сеть из тысяч кэширующих серверов (Edge-серверов), разбросанных по всему миру.

* **Как это работает:** Когда австралиец запрашивает `image.jpg`, DNS направляет его не в Москву, а на ближайший Edge-сервер в Сиднее. Если картинка там есть, она отдается за 10 мс. Если нет — Edge-сервер один раз скачает ее из Москвы, сохранит у себя и будет отдавать всем последующим австралийцам моментально.
* **Push vs Pull стратегии:**
* **Pull (Вытягивание):** CDN изначально пустая. Данные загружаются на Edge-сервера только тогда, когда их кто-то впервые запрашивает (подходит для сайтов с кучей мелких картинок, где не всё популярно).
* **Push (Проталкивание):** Ты сам принудительно заливаешь файлы на сервера CDN до того, как придут пользователи. (Подходит для релизов новых игр или выхода новой серии на Netflix — мы точно знаем, что через час этот файл запросят миллионы).



#### 3. Объектные хранилища (Object Storage / Amazon S3)

Куда вообще сохранять файлы, которые загружают пользователи?

* *В базу данных (PostgreSQL)?* Плохая идея. Файлы раздуют базу до неадекватных размеров, резервное копирование будет длиться сутками, а кэш оперативной памяти БД будет вымываться мусором.
* *На жесткий диск сервера приложения?* Тоже нельзя. Вспоминаем Главу 4: сервера должны быть **Stateless**. Если пользователь загрузил аватарку на Сервер №1, а следующий запрос пойдет на Сервер №2, аватарка там не найдется.

Решение — **Object Storage** (стандартом де-факто стал Amazon S3 API, который поддерживают все облака, включая Yandex и MinIO).

* В S3 нет привычных папок и файловой системы. Это плоское хранилище ключ-значение (как гигантский Redis на дисках), где Ключ — это путь (например, `users/avatars/123.jpg`), а Значение — сам файл (Blob — Binary Large Object).
* S3 бесконечно масштабируется и по умолчанию реплицирует каждый файл на 3 разных физических дата-центра (невероятная надежность).

#### 4. Супер-паттерн: Pre-signed URLs (Предварительно подписанные ссылки)

Это то, что обязательно спросят на System Design интервью при проектировании Instagram, YouTube или Telegram.

**Проблема:** Пользователь хочет загрузить видео на 1 ГБ. Если он отправит `POST` запрос с видео на твой бэкенд (Java/Spring), твой сервер будет занят приемом этого файла несколько минут. У тебя быстро закончатся потоки и память (OOM). Твой бэкенд не должен проксировать через себя тяжелые файлы!

**Решение (Pre-signed URL):**

1. Клиент (мобилка) делает легкий API запрос к твоему бэкенду: *"Хочу загрузить видео 'cat.mp4'"*.
2. Бэкенд проверяет права (авторизован ли пользователь) и идет в AWS S3. Он просит: *"Дай мне временный уникальный URL, по которому кто-то сможет положить 1 файл, и пусть URL сгорит через 15 минут"*.
3. Бэкенд возвращает этот сгенерированный (Pre-signed) URL клиенту.
4. Клиент начинает заливать гигабайтное видео **напрямую в S3**, минуя твой бэкенд. S3 спроектирован так, чтобы держать миллионы таких подключений одновременно.

**Пример на Java (Генерация Pre-signed URL для чтения):**

```java
import com.amazonaws.services.s3.AmazonS3;
import java.net.URL;
import java.util.Date;

public class S3Service {
    private final AmazonS3 s3Client;

    // Метод генерирует ссылку, по которой можно скачать приватный файл,
    // но ссылка будет работать только 1 час.
    public String generatePreSignedUrl(String bucketName, String objectKey) {
        // Устанавливаем срок годности (текущее время + 1 час)
        Date expiration = new Date();
        long expTimeMillis = expiration.getTime() + (1000 * 60 * 60);
        expiration.setTime(expTimeMillis);

        // Просим S3 сгенерировать подписанный URL с нашим ключом доступа
        URL url = s3Client.generatePresignedUrl(bucketName, objectKey, expiration);
        
        return url.toString();
    }
}

```

*Точно так же работает и загрузка (Upload), просто мы указываем метод `PUT` при генерации.*

#### 5. Тиринг хранения (Storage Tiering)

Хранить данные в горячем доступе (чтобы они отдавались за миллисекунды) — дорого. Поэтому S3-подобные хранилища делятся на уровни ("тиры"):

1. **Hot (Горячий):** Для файлов, которые читают прямо сейчас (новая аватарка, последний отчет). Дорогое хранение, но бесплатное/дешевое быстрое чтение.
2. **Warm (Теплый):** Для данных, к которым обращаются редко (фотографии пользователя пятилетней давности). Хранение дешевле, но за само чтение берут больше денег.
3. **Cold / Archive (Холодный - Glacier):** Для данных, которые, возможно, вообще никогда не понадобятся (логи аудита для налоговой за 2018 год). Хранение стоит копейки, но чтобы прочитать файл, придется отправить запрос и подождать от пары минут до 12 часов, пока роботы в дата-центре достанут нужную магнитную ленту.

---

**Итог:**

* **Reverse Proxy** защищает бэкенд, терминирует SSL и быстро отдает статику.
* **CDN** физически приближает тяжелый контент к пользователям по всему миру, сокращая задержки (Latency).
* **Object Storage (S3)** — идеальное место для хранения бесконечного количества файлов.
* **Pre-signed URLs** — критически важный паттерн, чтобы разгрузить сеть и CPU твоих серверов, заставляя клиентов скачивать и загружать файлы напрямую в S3.
* Управление стоимостью архивов решается через перенос старых файлов на **Cold (Холодные) уровни** хранения.
