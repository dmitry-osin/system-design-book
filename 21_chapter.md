**Глава 21. Хранилища специального назначения**

#### 1. Time-Series DB (Базы данных временных рядов)

**Проблема:** У тебя 10 000 серверов. Каждый сервер раз в секунду отправляет метрики: загрузка CPU, использование RAM, температура. Это 10 000 `INSERT` запросов в секунду. Через месяц это миллиарды строк. Обычная реляционная база сойдет с ума от перестроения B-Tree индексов.

**Решение:** Time-Series базы (TSDB), такие как **InfluxDB** или **Prometheus** (который мы упоминали в Главе 18).

* **Особенность:** Данные здесь всегда имеют одну структуру: `(Метка_времени, Источник, Значение)`. Данные **никогда не обновляются** (`UPDATE` не нужен) и почти никогда не удаляются поштучно. Они только дописываются в конец (Append-only).
* **Сжатие:** Поскольку значения температуры или CPU меняются плавно (например, 40.1, 40.2, 40.2), TSDB используют зверские алгоритмы сжатия (Gorilla compression). Они сохраняют только дельту (разницу) между соседними значениями, сжимая гигабайты метрик в мегабайты на диске.
* **Downsampling (Прореживание):** Никому не нужна температура сервера с точностью до секунды за прошлый год. TSDB умеют автоматически "схлопывать" старые данные. Данные за сегодня хранятся посекундно. Данные старше недели агрегируются до среднего за 1 час. Старше года — до 1 значения в день.

#### 2. Spatial DB (Пространственные БД и гео-поиск)

**Проблема:** Как работает Uber или Tinder? Тебе нужно найти "Ближайшие объекты в радиусе 5 км".
Если ты сохранишь в обычную таблицу координаты `latitude` (широта) и `longitude` (долгота), запрос будет выглядеть так:
`SELECT * FROM drivers WHERE lat BETWEEN x1 AND x2 AND lon BETWEEN y1 AND y2`.
Обычный B-Tree индекс базы данных **одномерный**. Он не может быстро искать пересечение сразу по двум осям. Базе придется перебрать миллионы строк (Full Scan).

**Решение:** Пространственные базы и индексы (например, расширение **PostGIS** для PostgreSQL).

Как они превращают 2D-координаты в одномерный индекс?

* **Geohash (Геохэш):** Карта мира делится на сетку квадратов. Каждому квадрату дается буквенный код. Весь мир — 'g'. Россия внутри 'g' — это 'v'. Москва внутри 'v' — это 'uc'. В итоге координата превращается в строку: `ucfu123`.
* *Магия:* Если два объекта находятся рядом на карте, их геохэши будут иметь **одинаковый префикс**! Поиск соседей превращается в молниеносный поиск по строке: `SELECT * FROM drivers WHERE geohash LIKE 'ucfu%'`.


* **Quadtree (Квадродерево):** Это структура данных в памяти. Мы берем карту города. Если в ней пусто — оставляем как есть. Если в квадрате больше 100 таксистов, мы делим его на 4 под-квадрата. Если в одном из под-квадратов всё еще тесно — делим его еще на 4. И так далее. Получается дерево, по которому можно за миллисекунды "спуститься" в нужный район и забрать водителей.

#### 3. Full-text Search (Полнотекстовый поиск)

**Проблема:** Если пользователь в интернет-магазине пишет "айфон дишево", обычный SQL запрос `WHERE description LIKE '%айфон дишево%'` не найдет ничего. SQL ищет точные совпадения и делает это чудовищно медленно, перечитывая все тексты на диске.

**Решение:** Поисковые движки **Elasticsearch** или **Apache Solr** (оба построены на базе библиотеки Apache Lucene).

Они используют структуру данных, которая называется **Inverted Index (Обратный индекс)**.
*Аналогия:* Оглавление в конце учебника по истории. Тебе не нужно читать всю книгу, чтобы найти главу про Наполеона. Ты открываешь букву "Н" в конце книги и видишь: *Наполеон — страницы 45, 112, 130*.

**Как работает Обратный индекс:**

1. **Анализ (Tokenization):** При добавлении описания товара "Купить дешево черный iPhone 15", Elasticsearch разбивает его на слова (токены).
2. **Стемминг (Нормализация):** Он приводит слова к базовой форме, убирая окончания и капитализацию (купить -> покупка, черный -> черн, iPhone -> iphone).
3. **Построение индекса:** Он создает словарь, где каждому слову соответствует список ID документов, где оно встречается.

*Пример Индекса:*

* `покупк` -> [Doc1, Doc5]
* `дешев` -> [Doc1, Doc2, Doc9]
* `iphone` -> [Doc1, Doc3]

Если пользователь ищет "дешевые покупки iphone", Elasticsearch мгновенно находит массивы `[Doc1, Doc2, Doc9]`, `[Doc1, Doc5]`, `[Doc1, Doc3]` и делает математическое пересечение. Ответ: **Doc1** (он встречается везде). При этом он вычисляет "релевантность" (TF-IDF), чтобы самый подходящий документ был на первом месте.

**Пример на Java (Поиск в Elasticsearch):**

```java
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch.core.SearchResponse;
import co.elastic.clients.elasticsearch.core.search.Hit;

public class SearchService {
    private final ElasticsearchClient esClient;

    public void searchProducts(String userInput) throws Exception {
        // Мы используем Fuzzy Query, чтобы прощать опечатки (например, "айфон" вместо "iphone")
        SearchResponse<Product> response = esClient.search(s -> s
            .index("products") // Название нашего индекса
            .query(q -> q
                .match(m -> m
                    .field("description")
                    .query(userInput)
                    .fuzziness("AUTO") // Автоматически исправляет 1-2 ошибки
                )
            ),
            Product.class
        );

        // Перебираем найденные хиты (результаты отсортированы по релевантности)
        for (Hit<Product> hit : response.hits().hits()) {
            System.out.println("Найден товар: " + hit.source().getName() 
                               + " с рейтингом совпадения: " + hit.score());
        }
    }
}

```

**Итог:**

* **Time-Series БД (InfluxDB/Prometheus)**: Идеальны для метрик. Быстро дописывают данные, сжимают их в десятки раз и автоматически "прореживают" исторические архивы.
* **Spatial БД (PostGIS)**: Необходимы для карт. Используют Геохэши или Quadtree, чтобы превратить 2D-кардинаты в одномерный поиск и мгновенно находить соседей на плоскости.
* **Full-text Search (Elasticsearch)**: Заменяют медленный SQL-поиск по тексту. Разбирают текст на слова, создают Обратный индекс (словарь) и позволяют искать с учетом опечаток, синонимов и морфологии языка за миллисекунды.
